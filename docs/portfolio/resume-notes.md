# Resume Notes

이 프로젝트는 "기술을 사용했다"가 아니라 "금융 거래 처리에서 어떤 리스크를 발견했고 어떻게 검증했는가"를 보여주는 용도로 정리한다.

자기소개서 문장은 다음 흐름을 따른다.

```txt
금융 IT에서 중요한 문제 인식
→ 판단 기준
→ 구현 액션
→ 검증
→ 배운 점
```

## 2. 대표 문장

### 정산 중복 방지

대용량 결제 데이터를 일별로 정산하는 배치 프로젝트를 구현하며, 정산 결과가 중복 저장되면 금액이 이중 반영될 수 있다는 점을 핵심 리스크로 보았습니다. 처음에는 정산일자 단위의 중복 방지만 고려했지만, 성능 비교를 위해 같은 날짜에 여러 처리 전략 결과를 저장해야 했습니다. 이에 중복 기준을 가맹점, 정산일자, 처리 전략 단위로 조정하고 DB unique constraint와 테스트로 검증했습니다.

### 실패 이력 추적

정산 배치에서는 실패 시 일부 결과만 저장되는 상황을 막는 것과 실패 원인을 추적하는 것이 모두 중요하다고 판단했습니다. Settlement 저장은 실패 시 롤백되도록 유지하고, BatchJobHistory는 별도 트랜잭션으로 저장해 실패 상태와 원인을 보존했습니다. 이를 통해 데이터 정합성과 운영 추적 가능성을 함께 고려하는 경험을 했습니다.

### GROUP_BY_QUERY 성능 개선

BASIC_LOOP 방식은 Payment 전체 Entity를 애플리케이션으로 조회해 Java에서 집계하는 기준선이었고, 데이터가 증가할수록 병목이 될 수 있다고 판단했습니다. DB GROUP BY로 가맹점별 결제금액과 취소금액을 먼저 집계하도록 개선하고, Java에서는 집계 결과만 받아 같은 계산식으로 정산금액을 산출했습니다. 개선 후에도 처리 전략별 정산 결과가 동일한지 자동 테스트와 DB 합계 검증으로 확인했습니다.

### 성능 수치에 대한 태도

성능 측정 결과는 단일 실행값으로 과장하지 않고, 같은 데이터와 같은 정산일자 기준에서 반복 측정 범위로 기록했습니다. 로컬 단일 DB에서는 캐시와 JVM warm-up의 영향을 받을 수 있으므로, 측정 조건과 한계를 함께 남기는 것이 신뢰할 수 있는 포트폴리오 기록이라고 판단했습니다.

## 3. 자기소개서 단락 예시

금융 IT에서 중요한 것은 단순히 기능이 동작하는 것이 아니라 금액 데이터가 일관되게 처리되고, 실패 원인이 추적 가능하며, 성능 개선 후에도 정합성이 유지되는 것이라고 생각합니다. 이를 확인하기 위해 대용량 결제 데이터를 일별로 정산하는 배치 프로젝트를 구현했습니다. BASIC_LOOP 기준선을 먼저 만들고, DB GROUP BY 집계를 적용한 GROUP_BY_QUERY 전략을 추가해 처리 시간을 비교했습니다. 이 과정에서 중복 정산 방지, 실패 시 Settlement 롤백과 BatchJobHistory 보존, 전략별 결과 동일성 테스트를 함께 구현했습니다. 성능 수치는 로컬 환경의 캐시 영향을 고려해 반복 측정 범위로 기록하며 과장하지 않는 태도를 유지했습니다.

## 4. 문장 조합용 키워드

- 정산 결과가 중복 저장되면 금액이 이중 반영될 수 있다고 판단했습니다.
- 실패한 배치의 원인을 추적할 수 있어야 운영 대응이 가능하다고 보았습니다.
- 성능 개선 후에도 총 결제금액, 총 취소금액, 수수료, 최종 정산금액이 동일한지 검증했습니다.
- DB가 잘하는 집계는 DB에서 수행하고, Java는 집계 결과를 바탕으로 정산 계산을 담당하도록 역할을 나눴습니다.
- 로컬 성능 수치는 단일값이 아니라 반복 측정 범위와 조건을 함께 기록했습니다.
