## 2. 현재 포트폴리오 주제

### 금융 거래 정산 배치 성능 개선

대용량 결제/취소 데이터를 일자별로 정산하는 과정에서 다음 문제를 다룬다.

- 같은 정산일자와 같은 전략이 중복 실행되면 정산 금액이 중복 반영될 수 있다.
- 배치가 실패했을 때 결과 데이터는 롤백되어야 하지만 실패 원인은 추적 가능해야 한다.
- 성능 개선 후에도 BASIC_LOOP와 GROUP_BY_QUERY의 정산 결과 금액이 같아야 한다.
- 대용량 데이터에서는 Payment 전체 Entity 로딩보다 DB GROUP BY 집계가 더 적절할 수 있다.

현재 구현 기준:

- Payment 100,000건
- Merchant 100개
- BASIC_LOOP 기준선 구현
- GROUP_BY_QUERY 개선 전략 구현
- BatchJobHistory 기반 실행 시간과 실패 원인 기록
- 중복 정산 방지
- 실패 시 Settlement 롤백, BatchJobHistory 보존

## 4. README와 연결되는 설명

README는 실행 방법과 측정 결과를 보여주는 공개 문서이고, 이 디렉터리는 그 내용을 자기소개서와 면접 답변으로 바꾸기 위한 원천 기록이다.

README에는 다음 흐름을 유지한다.

```txt
프로젝트 개요
→ 데이터 규모
→ BASIC_LOOP 기준선
→ GROUP_BY_QUERY 개선
→ GROUP_BY_BULK_SAVE 1차 저장 개선
→ 성능 비교
→ 100만/1000만 건 확장 실험 계획
→ 정합성/안정성 검증
→ 다음 개선 방향
```

현재 10만 건 / Merchant 100개 기준 실험은 완료된 기준 실험으로 관리한다. 측정값은 BASIC_LOOP 882ms, GROUP_BY_QUERY 133ms, GROUP_BY_BULK_SAVE 110ms이며, 이 결과는 Payment 전체 조회 방식의 한계와 DB GROUP BY 집계의 개선 효과를 설명하는 근거로 사용한다.

다만 Merchant 수가 100개라 Settlement 저장 결과가 100건 수준에 그쳤기 때문에 saveAll 기반 저장 최적화 효과는 제한적으로만 해석한다. 다음 실험은 100만 건 / Merchant 5,000개, 이후 1000만 건 / Merchant 5,000~10,000개로 확장해 조회 병목과 저장 병목을 분리해서 확인한다.

이 디렉터리에서는 같은 내용을 다음 관점으로 재정리한다.

```txt
문제 상황
→ 금융 관점의 중요성
→ 판단 기준
→ 기술 선택
→ 구현 액션
→ 검증
→ 포트폴리오/자소서/면접 문장
```

## 5. 작성 원칙

- 검증하지 않은 성능 수치를 쓰지 않는다.
- 수치는 단일 실행값보다 반복 측정 범위와 조건을 함께 적는다.
- 금융 IT 관점의 위험을 먼저 설명한 뒤 기술 선택을 설명한다.
- "Spring Boot를 사용했다"보다 "정산 실패 시 결과는 롤백하고 이력은 보존했다"처럼 문제 해결 관점으로 쓴다.
- 아직 구현하지 않은 기능은 개선 방향으로만 적는다.
