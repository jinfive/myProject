# Reliability

## 1. 목적

이 문서는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 안정성 기준을 정의한다.

이 프로젝트는 단순히 기능이 동작하는 것을 목표로 하지 않는다.

포트폴리오, PPT, 자기소개서, 면접에서 다음 흐름을 설명할 수 있어야 한다.

```txt
어떤 문제가 있었는가
→ 그 문제를 왜 중요하게 봤는가
→ 해결을 위해 어떤 판단을 했는가
→ 실제로 어떤 액션을 했는가
→ 어떻게 검증했는가
→ 결과를 어떻게 설명할 것인가
```

따라서 이 문서에서 말하는 안정성은 단순히 “에러가 나지 않는 것”이 아니다.

이 프로젝트에서 안정성은 다음을 의미한다.

- 같은 입력에 대해 같은 결과가 나온다.
- 정산 결과가 원천 데이터와 일치한다.
- 배치 실패 시 원인을 추적할 수 있다.
- 중복 실행으로 데이터가 깨지지 않는다.
- 성능 개선 후에도 계산 결과가 틀어지지 않는다.
- 외부 시세 API가 실패해도 2단계 실시간 처리 흐름이 완전히 중단되지 않는다.
- 주문·체결 결과가 잔고와 예수금에 정합성 있게 반영된다.
- README, PPT, 자소서에서 안정성 확보 액션을 설명할 수 있다.

---

## 2. 프로젝트 단계 기준

이 프로젝트는 하나의 저장소 안에서 두 단계로 진행한다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

현재는 1단계인 **정산 배치 성능 비교용 MVP 중 GROUP_BY_BULK_SAVE 1차 구현 완료** 단계이다.

현재 구현된 기능:

- Spring Boot 백엔드 구조 구성
- Merchant, Payment, Settlement, BatchJobHistory 도메인 구현
- 더미 데이터 자동 생성
    - 가맹점 100개
    - 결제 데이터 100,000건
    - 특정일 거래 약 70,000건
- BASIC_LOOP 정산 배치 구현
- GROUP_BY_QUERY 정산 배치 구현
- GROUP_BY_BULK_SAVE 1차 정산 배치 구현
- 정산 실행 API 구현
    - `POST /api/settlements/run`
- 정산 결과 조회 API 구현
    - `GET /api/settlements`
- 배치 이력 조회 API 구현
    - `GET /api/batch-histories`
- React 대시보드 구현
- 프론트 빌드 성공
- 백엔드 테스트 명령 성공

현재 안정성 리스크:

- `application.yml`과 `compose.yaml`의 DB 설정 불일치
- 정산 계산 테스트 부족
- 중복 정산 방지 테스트 부족
- 벌크 저장, 인덱스 적용 후 결과 정합성 검증 미구현
- 실패 배치 이력 저장 부족
- 재실행 정책 부족
- 대사 기능 미구현
- 실제 테이블 기준 DB 문서 미갱신

## 3. 안정성 목표

이 프로젝트는 다음 안정성 목표를 따른다.

### 3.1 1단계 배치 정산 안정성 목표

- 정산 배치는 같은 정산일자와 같은 데이터 기준으로 일관된 결과를 만들어야 한다.
- 처리 전략이 달라도 최종 정산금액은 동일해야 한다.
- 배치 실행 시작과 종료 이력이 남아야 한다.
- 배치 실패 시 실패 상태와 원인이 기록되어야 한다.
- 중복 정산으로 결과가 중복 저장되지 않아야 한다.
- 재실행 정책이 명확해야 한다.
- 정산 결과와 원천 Payment 데이터가 대사 가능해야 한다.
- 성능 개선 후에도 정산 계산 결과가 틀어지지 않아야 한다.
- DB 설정 불일치로 실행 환경이 흔들리지 않아야 한다.

---

### 3.2 2단계 증권 실시간 처리 안정성 목표

2단계는 1단계 완료 후 진행한다.

2단계의 안정성 목표는 다음과 같다.

- 실시간 시세 API가 실패해도 모의 시세로 대체할 수 있어야 한다.
- 주문은 접수 순서와 체결 우선순위에 따라 일관되게 처리되어야 한다.
- 같은 종목의 주문은 가격 우선·시간 우선 원칙을 따라야 한다.
- 체결 결과는 주문 상태, 보유 수량, 예수금에 함께 반영되어야 한다.
- 부분체결, 전체체결, 미체결 상태가 명확해야 한다.
- 예수금 부족 주문은 거절되어야 한다.
- 보유 수량 부족 매도 주문은 거절되어야 한다.
- WebSocket 연결 실패나 끊김 상황을 처리해야 한다.
- 초당 주문 수 증가 상황에서도 Queue 적체와 처리 지연을 확인할 수 있어야 한다.

## 5. 주요 실패 방식

## 5.1 배치 실행 실패

정산 배치는 다음 방식으로 실패할 수 있다.

- 정산 대상 데이터 조회 실패
- 가맹점 수수료율 조회 실패
- 정산 계산 중 오류 발생
- Settlement 저장 실패
- BatchJobHistory 저장 실패
- 중복 정산으로 인한 저장 실패
- DB 연결 실패
- 트랜잭션 처리 오류
- 대용량 조회로 인한 시간 지연 또는 메모리 부족

대응 기준:

- 배치 실행 시작 이력은 남겨야 한다.
- 실패 시 BatchJobHistory 상태를 `FAILED`로 기록한다.
- 실패 원인을 `error_message` 또는 `SettlementError`에 기록한다.
- 실패 후 재실행 가능 여부를 판단할 수 있어야 한다.
- BatchJobHistory 상태 check constraint는 `RUNNING`, `SUCCESS`, `FAILED`를 모두 허용해야 한다.
- 기존 로컬 DB의 check constraint가 새 enum 값을 허용하지 않는 경우 이력을 삭제하지 않고 제약조건만 갱신한다.

---

## 5.1.1 로컬 CORS 실패

로컬 개발에서 Vite 프론트엔드는 `http://localhost:5173`, Spring Boot 백엔드는 `http://localhost:8080`으로 실행된다.

정산 결과 초기화는 `DELETE /api/settlements?date=...` 요청을 사용하므로 브라우저가 OPTIONS preflight를 먼저 보낸다.

대응 기준:

- `/api/**` CORS 허용 origin은 로컬 개발 주소로 제한한다.
- `DELETE`와 `OPTIONS`를 허용 메서드에 포함한다.
- CORS 설정으로 해결하되, 로그인/JWT/Spring Security 구조를 불필요하게 추가하지 않는다.
- 정산 결과 초기화는 `settlements`만 삭제하고 `payments`, `merchants`, `batch_job_histories`는 보존한다.

---

## 5.2 데이터 정합성 실패

정산 데이터는 다음 방식으로 정합성이 깨질 수 있다.

- 같은 가맹점과 같은 정산일자가 중복 저장됨
- Payment 기준 총 결제금액과 Settlement 기준 총 결제금액이 다름
- Payment 기준 총 취소금액과 Settlement 기준 총 취소금액이 다름
- 수수료율 적용이 잘못됨
- 처리 전략별 결과가 서로 다름
- 배치 이력의 처리 건수와 실제 결과 건수가 다름

대응 기준:

- 중복 정산 방지 정책을 둔다.
- 정산 결과와 원천 데이터를 대사한다.
- 처리 전략이 달라도 결과 금액이 같은지 검증한다.
- 핵심 계산 로직 테스트를 추가한다.

---

## 5.3 성능 측정 실패

성능 개선 프로젝트에서는 성능 측정 자체도 안정적으로 관리해야 한다.

실패 방식:

- 데이터 건수가 다른 상태로 비교함
- 정산일자가 다른 상태로 비교함
- DB 설정이 다른 상태로 비교함
- 인덱스 적용 전후 조건이 다름
- 실행 시간이 기록되지 않음
- README와 실제 측정 결과가 다름

대응 기준:

- 같은 데이터 건수와 같은 정산일자로 비교한다.
- 같은 로컬 환경과 같은 DB 설정에서 비교한다.
- BatchJobHistory에 실행 시간을 기록한다.
- README와 PPT에는 실제 측정값만 반영한다.
- 로컬 벤치마크에서는 기존 Payment를 재사용하고 날짜만 오늘로 벌크 동기화해 실험 조건을 유지한다.
- Payment 날짜를 동기화할 때는 원천 데이터와 맞지 않는 기존 Settlement를 삭제하고, BatchJobHistory는 실행 이력이므로 보존한다.
- 날짜 동기화 기능은 운영 기능이 아니라 개발용 기능이며 설정으로 비활성화할 수 있어야 한다.

## 5.4 외부 시세 API 실패

2단계에서 실시간 주식 시세 API는 항상 실패할 수 있다고 가정한다.

실패 방식:

- API 서버 응답 없음
- API 인증 실패
- 응답 형식 변경
- WebSocket 연결 끊김
- 시세 수신 지연
- 호출 제한 초과

대응 기준:

- 모의 시세 생성기를 대체 흐름으로 둔다.
- 시세 출처를 `API` 또는 `MOCK`으로 구분한다.
- API 실패 이력을 남긴다.
- 실시간 주문·체결 흐름이 외부 API 하나에 완전히 의존하지 않게 한다.

---

## 5.5 주문·체결 실패

2단계 주문·체결 처리는 다음 방식으로 실패할 수 있다.

- 예수금 부족 주문이 접수됨
- 보유 수량 부족 매도 주문이 접수됨
- 주문이 Queue에 들어갔지만 처리되지 않음
- OrderBook 정렬 기준이 잘못됨
- 체결 수량이 주문 수량을 초과함
- 부분체결 후 잔여 수량이 잘못 계산됨
- 체결 내역은 저장됐지만 잔고가 반영되지 않음
- 예수금은 차감됐지만 체결 내역이 없음
- WebSocket 알림만 가고 DB 상태가 반영되지 않음

대응 기준:

- 주문 접수 전 예수금과 보유 수량을 검증한다.
- 체결 결과 저장, 주문 상태 변경, 잔고/예수금 변경은 하나의 트랜잭션 흐름으로 관리한다.
- 체결 수량, 잔여 수량, 주문 상태를 테스트한다.
- WebSocket 알림은 DB 반영 이후 발행한다.

## 6. 안정성 원칙

기본 안정성 원칙은 다음과 같다.

- 실패 가능성을 전제로 설계한다.
- 중요한 작업은 트랜잭션 경계를 명확히 한다.
- 중복 실행에 안전한 구조를 고려한다.
- 에러는 숨기지 않고 기록한다.
- 배치 실행 이력은 성공과 실패 여부와 관계없이 남긴다.
- 성능 개선 후에도 결과 정합성을 검증한다.
- 외부 서비스 호출에는 대체 흐름을 둔다.
- 사용자에게 이해 가능한 에러 메시지를 제공한다.
- 복구하기 어려운 변경은 작게 나누어 진행한다.
- 배포 전 검증 방법을 명확히 한다.
- README, PPT, 자소서에서 설명 가능한 안정성 확보 액션을 남긴다.

## 7. 트랜잭션과 데이터 정합성

데이터 변경 작업은 정합성을 유지해야 한다.

## 7.1 1단계 배치 정산 트랜잭션 원칙

정산 배치에서 검토할 작업:

- BatchJobHistory 시작 이력 생성
- Payment 조회
- Settlement 생성
- Settlement 저장
- SettlementError 저장
- ReconciliationResult 저장
- BatchJobHistory 종료 이력 업데이트

원칙:

- 같은 가맹점, 같은 정산일자, 같은 처리 전략의 중복 저장을 막는다.
- 정산 결과 저장과 배치 이력 업데이트의 관계를 명확히 한다.
- 배치 실패 시에도 실행 이력은 남아야 한다.
- 특정 가맹점만 실패한 경우 부분 실패 정책을 검토한다.
- 재실행 시 기존 결과를 삭제할지, 덮어쓸지, 거부할지 정책을 정한다.

---

## 7.2 2단계 주문·체결 트랜잭션 원칙

체결이 발생하면 다음 작업이 함께 처리되어야 한다.

```txt
Execution 저장
매수 주문 상태 변경
매도 주문 상태 변경
매수자 보유 수량 증가
매수자 예수금 감소
매도자 보유 수량 감소
매도자 예수금 증가
AccountTransaction 저장
ExecutionEvent 발행
```

원칙:

- 체결 결과와 잔고/예수금 변경은 분리되어서는 안 된다.
- 체결 수량은 매수/매도 주문의 잔여 수량을 초과할 수 없다.
- 부분체결 후 잔여 수량이 정확해야 한다.
- 체결 완료 후 주문 상태가 정확해야 한다.
- WebSocket 알림은 DB 상태 반영 이후 발행한다.

## 8. 재시도와 멱등성

재시도는 장애 대응에 도움이 될 수 있지만, 잘못 사용하면 중복 처리 문제를 만든다.

## 8.1 1단계 배치 재실행 기준

정산 배치는 재실행 시 중복 정산 위험이 있다.

따라서 다음 정책 중 하나를 명확히 선택해야 한다.

```txt
1. 같은 정산일자와 같은 전략은 재실행 거부
2. 기존 결과 삭제 후 재생성
3. 기존 결과를 덮어쓰기
4. 실패 건만 재처리
```

권장 방향:

- 초기 MVP에서는 같은 조건의 중복 실행을 거부한다.
- 이후 재실행 기능이 필요하면 기존 결과 삭제 후 재생성 또는 실패 건 재처리를 검토한다.
- 성능 비교를 위해 `merchant_id + settlement_date + processing_strategy` 기준으로 중복을 관리한다.
- 중복 여부는 실패 이력이 아니라 Settlement 결과 존재 여부를 기준으로 판단한다.
- 배치 이력은 정산 트랜잭션과 분리해 `RUNNING`, `SUCCESS`, `FAILED`와 실패 원인을 남긴다.
- 정산 처리 중 실패하면 Settlement 결과는 전체 롤백하고, BatchJobHistory의 실패 이력은 보존한다.

---

## 8.2 2단계 주문 재시도 기준

주문·체결에서 재시도는 매우 조심해야 한다.

원칙:

- 같은 주문이 중복 접수되지 않도록 주문 ID 또는 요청 ID를 관리한다.
- 체결 처리는 중복 실행되면 안 된다.
- WebSocket 알림 재전송은 가능하지만 DB 체결 결과를 중복 생성하면 안 된다.
- 주문 Queue 재처리 시 이미 처리된 주문인지 확인한다.

## 9. 외부 서비스 호출

## 17. 성능과 용량

성능 문제는 안정성 문제로 이어질 수 있다.

## 17.1 1단계 성능 안정성

성능 비교 기준:

```txt
BASIC_LOOP
GROUP_BY_QUERY
GROUP_BY_BULK_SAVE
GROUP_BY_BULK_INDEX
```

원칙:

- 같은 데이터 건수로 비교한다.
- 같은 정산일자로 비교한다.
- BASIC_LOOP는 실무 적용 방식이 아니라 성능 개선 전 기준선을 만들기 위해 의도적으로 구현한 방식이다.
- 실무 정산 배치라면 전체 Payment 데이터를 애플리케이션으로 가져오는 방식보다 DB GROUP BY 집계가 더 적절하다.
- GROUP_BY_QUERY는 Payment 전체 Entity 목록을 애플리케이션으로 로딩하지 않고, DB GROUP BY 결과만 받아 BASIC_LOOP와 같은 계산식과 반올림 정책으로 정산금액을 계산한다.
- GROUP_BY_QUERY 단계에서는 개선 효과를 분리하기 위해 Settlement 저장 방식을 개별 저장으로 유지한다.
- GROUP_BY_BULK_SAVE 1차 단계에서는 GROUP_BY_QUERY의 DB GROUP BY 집계 결과를 재사용하고, Settlement 저장 방식만 `saveAll` 기반으로 변경한다.
- 저장 방식 변경 후에는 집계 결과 건수, 생성 Settlement 수, 실제 저장 Settlement 수가 일치하는지 검증한다.
- BASIC_LOOP, GROUP_BY_QUERY, GROUP_BY_BULK_SAVE의 가맹점별 금액이 동일한지 검증한다.
- 10만 건 / Merchant 100개 실험에서는 Settlement 저장 결과가 100건 수준이므로 saveAll 기반 저장 최적화 효과를 제한적으로만 해석한다.
- 100만 건 / Merchant 5,000개 benchmark-medium 실험에서는 1000만 건으로 가기 전 정합성, 실행 시간, 메모리 부담, 저장 건수 증가 영향을 점검한다.
- benchmark-medium 데이터셋은 기존 benchmark 데이터에 추가하지 않고 `settlements`, `payments`, `merchants`를 초기화한 뒤 재생성한다.
- benchmark-medium 재생성 시에도 `batch_job_histories`는 실행 이력이므로 보존한다.
- 100만 건 기준 측정에서는 BASIC_LOOP 8,253ms, GROUP_BY_QUERY 899ms, GROUP_BY_BULK_SAVE 798ms로 기록되었고, 전략별 Settlement 5,000건과 합계 금액이 동일했다.
- 1000만 건 / Merchant 5,000~10,000개 실험에서는 GROUP_BY_QUERY와 GROUP_BY_BULK_SAVE 중심으로 대용량 정산 안정성과 실행 시간을 확인한다.
- `saveAll`만으로 실제 DB batch insert 효과를 단정하지 않고, Hibernate `batch_size`와 PostgreSQL `reWriteBatchedInserts=true`는 저장 병목이 확인된 뒤 분리 검토한다.
- 인덱스는 1000만 건에서 GROUP_BY_QUERY가 수 초 이상 걸리거나 `EXPLAIN ANALYZE`에서 `payments` 전체 스캔 비용이 크다고 확인될 때 적용을 검토한다.
- Spring Batch는 1000만 건 실험 이후 단순 속도보다 Job/Step 이력, chunk 처리, 실패 지점 추적, 재시작 가능성 같은 운영성 관점에서 도입 여부를 판단한다.
- 로컬 벤치마크 데이터 날짜 동기화는 Payment 전체를 Entity List로 로딩하지 않고 벌크 update/delete로 처리한다.
- 같은 DB 설정에서 비교한다.
- 실행 시간을 BatchJobHistory에 기록한다.
- 성능 개선 후에도 정산 결과가 동일한지 검증한다.
- README와 PPT에는 실제 측정 결과를 기록한다.

- WebSocket 알림 전송 건수

원칙:

- 실제 사용자가 없어도 VirtualOrderGenerator로 대량 주문 상황을 만든다.
- 외부 API 실패와 무관하게 모의 시세로 성능 테스트가 가능해야 한다.
- 성능 개선 후에도 체결 결과와 잔고/예수금 정합성이 깨지지 않아야 한다.

## 22. 핵심 원칙 요약

- 안정성은 단순히 에러가 없는 것이 아니다.
- 같은 입력에 대해 같은 결과가 나와야 한다.
- 정산 결과는 원천 데이터와 대사 가능해야 한다.
- 배치 실행 이력은 성공과 실패 여부와 관계없이 남아야 한다.
- 실패 원인은 추적 가능해야 한다.
- 재실행과 중복 실행 정책은 명확해야 한다.
- 성능 개선 후에도 데이터 정합성이 유지되어야 한다.
- 외부 시세 API는 실패할 수 있으므로 모의 시세 대체 흐름을 둔다.
- 주문·체결 결과는 잔고와 예수금에 정합성 있게 반영되어야 한다.
- WebSocket 알림은 DB 상태 반영 이후 전송한다.
- 현재는 1단계 배치 성능 개선을 우선한다.
- 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 확정한다.
- 결과 중심이 아니라 액션 중심으로 안정성 확보 과정을 기록한다.
