
# Quality Score

## 1. 목적

이 문서는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 품질 평가 기준을 정의한다.

이 프로젝트는 단순히 기능이 동작하는 것을 목표로 하지 않는다.

포트폴리오, PPT, 자기소개서, 면접에서 다음 흐름을 설명할 수 있어야 한다.

```txt
어떤 문제가 있었는가
→ 그 문제를 왜 중요하게 봤는가
→ 해결을 위해 어떤 판단을 했는가
→ 실제로 어떤 액션을 했는가
→ 어떻게 검증했는가
→ 결과를 어떻게 설명할 것인가
````

따라서 품질 평가는 단순 코드 품질뿐 아니라 다음 항목까지 함께 평가한다.

* 정산 결과의 정확성
* 데이터 정합성
* 배치 실행 안정성
* 성능 개선 검증 가능성
* 실시간 처리 확장 가능성
* 테스트 신뢰도
* 문서와 코드의 일치 여부
* README, PPT, 자소서 설명 가능성

---

## 2. 프로젝트 단계 기준

이 프로젝트는 하나의 저장소 안에서 두 단계로 진행한다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

현재는 1단계인 **정산 배치 성능 비교용 MVP 중 GROUP_BY_BULK_SAVE 1차 구현 완료** 단계이다.

현재 구현된 기능:

* Spring Boot 백엔드 구조 구성
* Merchant, Payment, Settlement, BatchJobHistory 도메인 구현
* 더미 데이터 자동 생성

    * 가맹점 100개
    * 결제 데이터 100,000건
    * 특정일 거래 약 70,000건
* BASIC_LOOP 정산 배치 구현
* GROUP_BY_QUERY 정산 배치 구현
* GROUP_BY_BULK_SAVE 1차 정산 배치 구현
* 정산 실행 API 구현

    * `POST /api/settlements/run`
* 정산 결과 조회 API 구현

    * `GET /api/settlements`
* 배치 이력 조회 API 구현

    * `GET /api/batch-histories`
* React 대시보드 구현
* 프론트 빌드 성공
* 백엔드 테스트 명령 성공

현재 부족한 부분:

* DB 설정 불일치
* 정산 계산 테스트 부족
* 중복 정산 방지 테스트 부족
* 벌크 저장 최적화 미구현
* 인덱스 적용 및 성능 비교 미구현
* 실패 이력 저장 부족
* 재실행 정책 부족
* 대사 기능 미구현
* 성능 비교 결과 문서화 부족
* PPT용 설명 구조 부족

## 3. 품질 평가 원칙

좋은 변경은 다음 조건을 만족해야 한다.

* 기존 기능을 깨지 않는다.
* 변경 범위가 명확하다.
* 코드가 읽기 쉽다.
* 책임이 적절히 분리되어 있다.
* 정산 결과가 정확하다.
* 데이터 정합성을 해치지 않는다.
* 실패 상황을 추적할 수 있다.
* 성능 개선 전후를 비교할 수 있다.
* 테스트 또는 수동 검증 방법이 있다.
* 필요한 문서가 갱신되어 있다.
* 불필요한 복잡성을 만들지 않는다.
* README, PPT, 자소서에서 설명할 수 있는 액션 중심 기록이 남아 있다.
* 로컬 벤치마크용 데이터 보정 기능은 운영 기능과 구분되고 설정으로 제어된다.

## 6. 세부 평가 기준

## 6.1 정확성

확인할 것:

* 정산 계산식이 올바른가
* 결제금액과 취소금액이 정확히 반영되는가
* 가맹점별 수수료율이 정확히 적용되는가
* 처리 전략이 달라도 정산 결과가 일관되는가
* API 응답이 기대한 형태로 반환되는가
* 기존 기능을 깨지 않는가
* Payment 날짜 동기화 시 기존 Settlement를 삭제해 원천 데이터와 결과 데이터 불일치를 막는가

정산 계산 기준:

```txt
순매출 = 총 결제금액 - 총 취소금액
수수료 = 순매출 × 가맹점 수수료율
정산금액 = 순매출 - 수수료
```

점수 기준:

| 점수 | 기준                           |
| -- | ---------------------------- |
| 5  | 정산 계산, API 응답, 예외 상황이 모두 정확함 |
| 4  | 핵심 계산은 정확하며 사소한 보완만 필요       |
| 3  | 기본 동작은 하지만 일부 예외 검증이 부족함     |
| 2  | 계산 또는 응답 일부가 불안정함            |
| 1  | 핵심 정산 기능이 올바르게 동작하지 않음       |

---

## 6.2 정합성

확인할 것:

* 같은 정산일자와 같은 가맹점의 중복 정산을 방지하는가
* Payment 기준 집계와 Settlement 결과가 일치하는가
* BatchJobHistory가 실제 배치 실행 결과와 일치하는가
* 실패가 발생해도 이력이 남는가
* 재실행 시 기존 결과와 충돌하지 않는가
* 금액 필드에 `double`, `float` 대신 `BigDecimal`을 사용하는가

2단계 증권 실시간 처리에서는 다음도 확인한다.

* 체결 결과와 주문 상태가 일치하는가
* 체결 수량과 잔여 수량이 일치하는가
* 매수 체결 시 예수금 감소와 보유 수량 증가가 함께 처리되는가
* 매도 체결 시 보유 수량 감소와 예수금 증가가 함께 처리되는가
* 잔고 부족, 보유 수량 부족 상황을 처리하는가

점수 기준:

| 점수 | 기준                          |
| -- | --------------------------- |
| 5  | 원천 데이터, 결과, 이력, 상태가 모두 일관됨  |
| 4  | 주요 정합성은 보장되며 일부 보완만 필요      |
| 3  | 기본 정합성은 있으나 테스트나 대사 기능이 부족함 |
| 2  | 중복 처리나 금액 불일치 가능성이 있음       |
| 1  | 데이터 정합성이 깨질 가능성이 큼          |

## 6.3 성능 개선성

확인할 것:

* 성능 병목을 명확히 정의했는가
* BASIC_LOOP 기준선이 있는가
* GROUP_BY_QUERY 개선 전후를 비교할 수 있는가
* GROUP_BY_BULK_SAVE 적용 전후를 비교할 수 있는가
* GROUP_BY_BULK_INDEX 적용 전후를 비교할 수 있는가
* 같은 데이터 건수와 같은 정산일자로 비교했는가
* BatchJobHistory에 실행 시간이 기록되는가
* README와 PPT에 성능 비교 결과를 반영할 수 있는가
* GROUP_BY_QUERY가 Payment 전체 Entity 목록이 아니라 DB GROUP BY 집계 결과만 조회하는가
* GROUP_BY_QUERY가 BASIC_LOOP와 동일한 금액 결과를 내는지 자동 테스트로 검증하는가
* GROUP_BY_BULK_SAVE가 Payment 전체 Entity 목록이 아니라 DB GROUP BY 집계 결과를 재사용하는가
* GROUP_BY_BULK_SAVE가 Settlement 개별 `save` 반복 대신 `saveAll`을 사용하는가
* GROUP_BY_BULK_SAVE 적용 후 집계 결과 건수, 생성 Settlement 수, 저장 Settlement 수가 일치하는가
* GROUP_BY_BULK_SAVE 적용 후 BASIC_LOOP, GROUP_BY_QUERY와 금액 결과가 동일한가

1단계 성능 비교 기준:

```txt
BASIC_LOOP
GROUP_BY_QUERY
GROUP_BY_BULK_SAVE
GROUP_BY_BULK_INDEX
```

2단계 성능 비교 기준:

```txt
DB_DIRECT
QUEUE_BASED
MEMORY_ORDER_BOOK
SYMBOL_PARTITIONED
```

점수 기준:

| 점수 | 기준                         |
| -- | -------------------------- |
| 5  | 병목, 개선 액션, 측정 결과가 모두 명확함   |
| 4  | 주요 개선 결과가 있으며 일부 보완만 필요    |
| 3  | 개선 시도는 있으나 비교 조건이나 수치가 부족함 |
| 2  | 성능 개선을 주장하지만 검증 근거가 약함     |
| 1  | 성능 개선 프로젝트로 보기 어려움         |

* 트랜잭션 경계가 적절한가
* 장애 발생 시 원인을 찾을 수 있는 로그나 이력이 있는가

현재 특히 확인할 항목:

```txt
application.yml과 compose.yaml DB 설정 일치 여부
배치 실패 이력 저장 여부
재실행 정책 존재 여부
중복 정산 방지 여부
```

```txt
시세 API 실패 시 모의 시세로 대체 가능한가
주문 Queue 장애 또는 지연 상황을 추적할 수 있는가
WebSocket 연결 끊김을 처리하는가
```

점수 기준:

| 점수 | 기준                      |
| -- | ----------------------- |
| 5  | 실패 상황과 복구 가능성이 충분히 고려됨  |
| 4  | 주요 실패 상황을 처리함           |
| 3  | 기본 처리는 있으나 일부 위험이 남아 있음 |
| 2  | 실패 상황 처리가 부족함           |
| 1  | 장애나 데이터 손상 가능성이 큼       |

## 13. 현재 품질 진단

현재 기준으로 프로젝트는 다음 상태이다.

```txt
정산 배치 성능 비교용 MVP 중 GROUP_BY_BULK_SAVE 1차 구현 완료
```

현재 장점:

* 기본 Spring Boot 백엔드 구조가 구성되어 있다.
* 핵심 도메인인 Merchant, Payment, Settlement, BatchJobHistory가 구현되어 있다.
* 대용량 더미 데이터가 생성된다.
* BASIC_LOOP 기준선이 구현되어 있다.
* GROUP_BY_QUERY 개선 전략이 구현되어 있다.
* 정산 실행과 조회 API가 동작한다.
* React 대시보드가 있다.
* 빌드와 기본 테스트 명령이 성공한다.

현재 품질 리스크:

* 정산 계산 테스트가 부족하다.
* 중복 정산 방지 테스트가 부족하다.
* DB 설정이 불일치한다.
* GROUP BY, 벌크 저장, 인덱스 적용이 아직 없어 성능 개선 결과가 부족하다.
* 실패 이력과 재실행 정책이 약하다.
* 대사 기능이 없어 정합성 검증 메시지가 약하다.
* DB 문서가 실제 코드와 맞지 않을 수 있다.
* PPT와 자소서용 액션 중심 설명 구조가 아직 부족하다.

현재 우선 개선 순서:

```txt
1. 10만 건 / Merchant 100개 기준 실험 정리
2. 100만 건 / Merchant 5,000개 중간 확장 실험
3. 1000만 건 / Merchant 5,000~10,000개 최종 대용량 실험
4. 저장 병목 확인 시 Hibernate batch_size 적용 검토
5. 저장 병목이 계속 남으면 PostgreSQL reWriteBatchedInserts 적용 검토
6. 1000만 건 조회 병목 확인 시 GROUP_BY_BULK_INDEX 구현
7. EXPLAIN ANALYZE로 실행 계획 확인
8. 대사 기능 추가
9. RUNNING 중복 실행 방지
```

## 14. 핵심 원칙 요약

* 품질은 “동작 여부”만으로 판단하지 않는다.
* 정산 결과의 정확성과 데이터 정합성을 우선한다.
* 성능 개선은 반드시 기준선과 비교 수치가 있어야 한다.
* 테스트 없는 성능 개선은 신뢰도가 낮다.
* 실패 이력과 재실행 정책은 안정성의 핵심이다.
* 문서와 코드가 다르면 문서를 갱신한다.
* 결과 중심이 아니라 액션 중심으로 기록한다.
* README, PPT, 자소서에서 설명 가능한 변경이 좋은 변경이다.
* 현재는 1단계 배치 성능 개선을 우선한다.
* 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 확정한다.
