# Plans

## 1. 목적

이 문서는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 계획, 로드맵, 현재 우선순위, 실행 중인 작업을 관리하기 위한 기준을 정의한다.

이 프로젝트는 단순 기능 구현 프로젝트가 아니다.

포트폴리오, PPT, 자기소개서, 면접 설명에서 다음 흐름을 보여주는 것이 핵심이다.

```txt
어떤 문제가 있었는가
→ 그 문제를 왜 중요하게 봤는가
→ 해결을 위해 어떤 생각을 했는가
→ 실제로 어떤 액션을 했는가
→ 어떻게 검증했는가
→ 결과를 어떻게 설명할 것인가
```

따라서 계획 문서는 단순 할 일 목록이 아니라, 다음 내용을 명확히 하기 위해 사용한다.

- 무엇을 만들 것인가
- 왜 필요한가
- 어떤 문제가 있는가
- 어떤 판단으로 해결 방향을 정했는가
- 어떤 범위까지 작업할 것인가
- 어떤 순서로 진행할 것인가
- 어떤 위험이 있는가
- 완료 기준은 무엇인가
- 검증 방법은 무엇인가
- README, PPT, 자소서에 어떻게 연결할 것인가
- 남은 기술부채는 무엇인가

## 2. 프로젝트 전체 방향

이 프로젝트는 하나의 저장소 안에서 두 단계로 진행한다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

### 2.1 1단계 목표

1단계는 금융권 공통 역량을 보여주기 위한 프로젝트이다.

핵심 키워드:

- 대용량 배치 처리
- 정산
- 대사
- 데이터 정합성
- 안정성
- 성능 개선
- 배치 실행 이력
- 실패 이력
- GROUP BY 개선
- 벌크 저장
- 인덱스 튜닝

현재 상태:

```txt
정산 배치 성능 비교용 MVP 중 BASIC_LOOP 기준선 구현 완료
```

현재 구현된 기능:

- Spring Boot 백엔드 구조 구성
- Merchant, Payment, Settlement, BatchJobHistory 도메인 구현
- 더미 데이터 자동 생성
    - 가맹점 100개
    - 결제 데이터 100,000건
    - 특정일 거래 약 70,000건
- BASIC_LOOP 정산 배치 구현
- 정산 실행 API 구현
    - POST /api/settlements/run
- 정산 결과 조회 API 구현
    - GET /api/settlements
- 배치 이력 조회 API 구현
    - GET /api/batch-histories
- React 대시보드 구현
- 프론트 빌드 성공
- 백엔드 테스트 명령 성공

현재 부족한 부분:

- DB 설정 불일치
- 정산 계산 테스트 부족
- 중복 정산 방지 테스트 부족
- GROUP BY 기반 개선 전략 미구현
- 벌크 저장 최적화 미구현
- 인덱스 적용 및 성능 비교 미구현
- 실패 이력 저장 부족
- 재실행 정책 부족
- 대사 기능 미구현
- 성능 비교 결과 문서화 부족
- PPT용 설명 구조 부족

현재 로컬 벤치마크 정책:

- 대용량 성능 비교를 반복하기 위해 기존 Payment 데이터를 재사용한다.
- 기존 Payment가 있으면 새로 insert하지 않고, 필요할 때 `transaction_date`만 오늘 날짜로 동기화한다.
- 날짜가 바뀌면 기존 Settlement는 원천 데이터와 맞지 않으므로 삭제한다.
- BatchJobHistory는 실행 이력이므로 삭제하지 않는다.
- 이 기능은 운영 기능이 아니라 개발용 기능이며 `benchmark.data-date-sync-enabled` 설정으로 제어한다.

---

### 2.2 2단계 목표

2단계는 증권사 특화 역량을 보여주기 위한 프로젝트이다.

단순히 “실시간 기능”을 추가하는 것이 아니라, 다음 흐름을 구현한다.

```txt
실시간 주식 시세 API 수신
→ 현재가 기반 모의 주문 생성
→ 주문 Queue 적재
→ 메모리 기반 OrderBook 처리
→ 가격 우선·시간 우선 체결
→ 주문 상태 변경
→ 잔고와 예수금 반영
→ 체결 결과 DB 저장
→ WebSocket 실시간 알림
→ 초당 주문 수 기준 성능 테스트
```

핵심 키워드:

- 실시간 시세 처리
- 모의 주문 생성
- 주문 Queue
- OrderBook
- Matching Engine
- 가격 우선·시간 우선 체결
- 부분체결
- 미체결
- 잔고 정합성
- 예수금 정합성
- WebSocket 알림
- 대량 주문 성능 테스트

2단계는 아직 구현하지 않는다.

현재는 1단계 배치 성능 개선 프로젝트를 먼저 완성한다.

## 3. 계획 문서의 역할

계획 문서는 다음 상황에서 사용한다.

- 새로운 핵심 기능을 추가할 때
- 기존 기능을 크게 수정할 때
- 아키텍처를 변경할 때
- 데이터베이스 구조를 변경할 때
- 성능 개선 전략을 추가할 때
- 배치 실행 흐름을 변경할 때
- 정산 계산식이나 대사 기준을 변경할 때
- 외부 주식 시세 API를 연동할 때
- 주문 Queue, OrderBook, Matching Engine을 추가할 때
- WebSocket 실시간 알림을 추가할 때
- 큰 리팩토링을 진행할 때
- 기술부채를 정리할 때
- README, PPT, 자소서용 설명 구조를 정리할 때

작은 오타 수정, 단순 스타일 수정, 명확한 버그 수정처럼 범위가 작은 작업에는 별도 계획 문서가 필요하지 않을 수 있다.

다만 작은 작업이라도 다음에 영향을 주면 실행 계획을 작성한다.

- DB 구조
- 정산 계산식
- 배치 실행 흐름
- 성능 측정 기준
- 대사 기준
- 실패 이력
- 재실행 정책
- 실시간 시세 API 흐름
- 모의 주문 생성 기준
- 주문 Queue 구조
- OrderBook 구조
- 체결 처리 흐름
- 잔고/예수금 정합성
- WebSocket 알림 구조

- 부분체결/전체체결/미체결 상태 추가
- 잔고와 예수금 반영 로직 추가

## 6. 계획이 필요하지 않을 수 있는 작업

다음 작업은 별도 계획 없이 진행할 수 있다.

- 오타 수정
- 문서 표현 개선
- 작은 UI 문구 수정
- 단순한 변수명 정리
- 명확한 버그 수정
- 기존 패턴을 그대로 따르는 작은 기능 추가
- 테스트 코드 일부 보완
- 표기 통일
- README 문장 일부 보완

단, 작은 작업이라도 안정성, 정합성, 성능 비교, DB 구조, 실시간 처리 흐름에 영향을 주면 계획을 작성한다.

## 7. 실행 계획 작성 원칙

이 프로젝트의 실행 계획은 단순히 “무엇을 구현한다”로 끝나지 않는다.

반드시 다음 흐름으로 작성한다.

```txt
문제
→ 판단
→ 액션
→ 검증
→ 결과 정리
```

예시:

```txt
문제:
BASIC_LOOP 방식은 특정 정산일자의 Payment 데이터를 모두 조회해 Java 반복문으로 집계한다.

판단:
데이터가 증가할수록 전체 Entity 로딩과 반복 집계가 병목이 될 수 있다.

액션:
PaymentRepository에 GROUP BY 집계 쿼리를 추가하고, 필요한 집계 결과만 Projection으로 조회한다.

검증:
동일한 정산일자와 동일한 데이터 건수로 BASIC_LOOP와 GROUP_BY_QUERY 처리 시간을 비교한다.

결과 정리:
README와 PPT에 처리 시간 비교표와 개선 이유를 정리한다.
```

이 원칙은 README, PPT, 자소서 작성 방향과도 동일하다.

## 9. 로드맵 작성 기준

로드맵은 장기적인 작업 방향을 정리할 때 사용한다.

로드맵은 다음 기준으로 작성한다.

- 현재 가장 중요한 문제를 먼저 다룬다.
- 1단계 배치 성능 개선을 먼저 완성한다.
- 2단계 증권 실시간 처리는 1단계 완료 후 진행한다.
- 핵심 기능과 정합성을 우선한다.
- 성능 개선은 반드시 수치로 검증한다.
- 있으면 좋은 기능보다 포트폴리오 핵심 메시지에 필요한 기능을 먼저 한다.
- 큰 작업은 여러 단계로 나눈다.
- 각 단계의 완료 기준을 명확히 한다.
- README와 PPT에 반영 가능한 결과를 남긴다.

## 현재 목표

현재 목표는 1단계 배치 성능 개선 프로젝트를 완성하는 것이다.

현재는 BASIC_LOOP 기준선만 구현되어 있으므로, 성능 개선 프로젝트로 보이기 위해서는 다음 개선 전략이 반드시 필요하다.

```txt
GROUP_BY_QUERY
GROUP_BY_BULK_SAVE
GROUP_BY_BULK_INDEX
```

또한 금융권 포트폴리오로 설득력을 갖기 위해 다음도 필요하다.

```txt
정산 계산 테스트
중복 정산 방지
배치 실패 이력
재실행 정책
대사 기능
성능 비교 결과 문서화
```

- 실제 테이블 기준 DB 문서 갱신
- 정산 처리 전략 구조 분리

## 17. 기술부채 관리

기술부채는 숨기지 않고 기록한다.

기술부채로 기록해야 하는 경우:

- 임시 구현이 들어간 경우
- 테스트 없이 기능을 먼저 구현한 경우
- 중복 코드가 많지만 즉시 정리하지 못한 경우
- 구조 개선이 필요하지만 범위가 큰 경우
- 성능 문제가 예상되지만 당장 장애는 아닌 경우
- 문서가 부족한 경우
- 정합성 검증이 부족한 경우
- 실패 이력 관리가 부족한 경우
- 포트폴리오에서 설명하기 어려운 구조가 남은 경우
- 추후 리팩토링이 필요한 경우

기술부채는 다음 문서에 기록한다.

```txt
docs/exec-plans/tech-debt-tracker.md
```

```md
## 기술부채 항목명
```

## 18. 완료된 계획 처리

작업이 완료되면 다음을 수행한다.

1. 실행 계획의 완료 기준을 확인한다.
2. 검증 결과를 기록한다.
3. 성능 측정 결과가 있으면 수치를 기록한다.
4. 정합성 검증 결과가 있으면 기록한다.
5. 남은 문제가 있으면 기술부채에 기록한다.
6. 관련 문서를 갱신한다.
7. README 또는 PPT 반영 포인트를 정리한다.
8. 계획 문서를 `active`에서 `completed`로 이동한다.

완료된 계획은 삭제하지 않는다.

나중에 의사결정 근거와 변경 이력을 확인하기 위해 보관한다.

## 19. 계획 작성 시 주의사항

계획은 너무 복잡하게 작성하지 않는다.

좋은 계획은 다음 조건을 만족한다.

- 목표가 명확하다.
- 작업 범위가 분명하다.
- 하지 않을 일이 적혀 있다.
- 문제 상황이 설명되어 있다.
- 판단 기준이 드러나 있다.
- 구현 액션이 구체적이다.
- 검증 방법이 있다.
- 위험 요소가 드러나 있다.
- 완료 기준이 확인 가능하다.
- README와 PPT에 반영할 포인트가 있다.

좋지 않은 계획:

- 목표가 모호하다.
- 작업 범위가 계속 커진다.
- 한 번에 너무 많은 것을 바꾸려 한다.
- 검증 방법이 없다.
- 실패했을 때 되돌릴 방법이 없다.
- “구현했다”만 있고 왜 했는지가 없다.
- 결과만 있고 액션이 없다.

## 20. 에이전트 작업 지침

AI 에이전트가 작업할 때는 다음을 따른다.

- 큰 작업은 먼저 실행 계획을 작성한다.
- 기존 계획이 있으면 그 계획을 우선 확인한다.
- 계획 없이 대규모 구조 변경을 하지 않는다.
- 작업 범위를 임의로 확장하지 않는다.
- 현재는 1단계 배치 성능 개선을 우선한다.
- 2단계 증권 실시간 처리는 기획이 확정된 확장 범위로 관리한다.
- 1단계 완료 전 2단계 구현을 무리하게 시작하지 않는다.
- DB, 아키텍처, 성능 측정, 대사 기준, 실시간 처리 흐름 변경은 별도 계획을 둔다.
- 진행 중 발견한 기술부채는 기록한다.
- 완료된 작업은 검증 방법과 결과를 남긴다.
- 계획과 실제 구현이 달라지면 계획 문서를 갱신한다.
- README, PPT, 자소서에서 설명할 수 있는 액션 중심 결과를 남긴다.

## 22. 핵심 원칙 요약

- 계획은 작업을 느리게 하기 위한 문서가 아니다.
- 계획은 작업 범위를 명확히 하기 위한 도구다.
- 큰 변경은 계획 없이 진행하지 않는다.
- 완료 기준이 없는 작업은 완료된 것이 아니다.
- 현재는 1단계 배치 성능 개선을 먼저 완성한다.
- 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 확정한다.
- 결과 중심이 아니라 액션 중심으로 계획한다.
- 문제, 판단, 액션, 검증, 결과 정리 흐름을 남긴다.
- 성능 개선은 반드시 수치로 검증한다.
- 정합성 검증은 테스트나 대사 결과로 남긴다.
- 기술부채는 숨기지 않고 기록한다.
- 계획은 실제 구현에 맞게 계속 갱신한다.
- README, PPT, 자소서에서 설명 가능한 작업 기록을 남긴다.
