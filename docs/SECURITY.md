# Security

## 1. 목적

이 문서는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 보안 기준을 정의한다.

이 프로젝트는 로그인, 회원가입, 권한 관리가 핵심인 서비스가 아니다.

프로젝트의 핵심은 다음 두 단계이다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

따라서 이 문서의 보안 기준은 복잡한 인증/인가 구현보다 다음 항목에 중점을 둔다.

- 입력값 검증
- API 안전성
- 민감정보와 시크릿 관리
- 로그와 에러 메시지 보안
- DB 접속 정보 보호
- 외부 주식 시세 API 키 보호
- 정산 데이터와 성능 측정 데이터 보호
- 모의 주문·체결 데이터 정합성 보호
- 금융권 포트폴리오로서 지켜야 할 기본 보안 태도

이 프로젝트는 토이 프로젝트이지만 금융권 제출과 면접 설명을 전제로 한다.

따라서 다음 원칙을 지킨다.

```txt
보안 기능을 과하게 만들지는 않는다.
하지만 금융 데이터를 다루는 프로젝트처럼 최소한의 안전한 기본값은 지킨다.
```

---

## 2. 프로젝트 보안 범위

## 2.1 보안의 목표

이 프로젝트에서 보안의 목표는 다음과 같다.

- 정산 실행 API가 잘못된 입력으로 오작동하지 않도록 한다.
- 정산일자, 처리 전략, 금액, 수량 등 입력값을 검증한다.
- DB 접속 정보와 외부 API 키를 코드에 하드코딩하지 않는다.
- 로그에 민감한 설정값이나 API 키를 남기지 않는다.
- 외부 주식 시세 API 응답을 그대로 신뢰하지 않는다.
- 프론트엔드에 비공개 시크릿을 노출하지 않는다.
- 에러 메시지에 내부 구조나 민감정보를 노출하지 않는다.
- 대량 더미 데이터 생성, 배치 재실행, 주문 생성기 실행처럼 위험한 기능은 통제한다.
- README, PPT, 자기소개서에서 “금융권 프로젝트로서 최소 보안 기준을 고려했다”고 설명할 수 있게 한다.

---

## 2.2 이 프로젝트에서 하지 않는 것

이 프로젝트는 다음 기능을 핵심 범위로 두지 않는다.

```txt
회원가입
로그인
JWT 인증
Refresh Token
권한 관리
관리자 권한 체계
비밀번호 재설정
소셜 로그인
실제 결제 승인
실제 증권 주문 API 호출
실제 고객 개인정보 관리
```

따라서 이 문서는 인증/인가 구현 방법을 자세히 다루지 않는다.

다만 향후 배포하거나 외부에 공개할 경우, 다음 API는 최소한의 접근 통제가 필요하다.

```txt
더미 데이터 생성 API
정산 배치 실행 API
정산 결과 삭제 또는 재실행 API
인덱스 적용 또는 DB 초기화 관련 API
모의 주문 생성기 시작/중지 API
```

## 3. 보안 모델

이 프로젝트는 다음 보안 원칙을 따른다.

- 외부 입력값은 신뢰하지 않는다.
- API 요청값은 DTO 또는 명확한 파라미터 기준으로 검증한다.
- 금액, 날짜, 수량, 처리 전략 값은 허용 범위를 검증한다.
- Entity를 API 응답으로 그대로 노출하지 않는다.
- API 응답에는 필요한 데이터만 포함한다.
- DB 접속 정보, API Key, Secret은 코드와 문서에 직접 작성하지 않는다.
- 로그에는 시크릿, API Key, Authorization Header, 전체 환경 변수를 남기지 않는다.
- 외부 시세 API 응답은 검증한 뒤 사용한다.
- 위험한 작업은 명확한 실행 조건과 확인 절차를 둔다.
- 보안 구조는 프로젝트 규모에 맞게 단순하게 유지한다.

---

## 4. 위협 가정

이 프로젝트에서 고려해야 할 위협은 다음과 같다.

## 4.1 1단계 배치 정산 관련 위협

- 잘못된 정산일자가 입력될 수 있다.
- 허용되지 않은 처리 전략 값이 전달될 수 있다.
- 정산 배치가 반복 호출되어 중복 정산이 발생할 수 있다.
- 더미 데이터 생성 API가 반복 호출되어 DB가 불필요하게 커질 수 있다.
- 정산 결과 조회 API가 과도한 데이터를 반환할 수 있다.
- 에러 메시지에 DB 구조나 내부 예외가 노출될 수 있다.
- 로그에 DB 접속 정보가 남을 수 있다.
- 성능 측정 결과가 실제 설정과 맞지 않게 기록될 수 있다.

---

## 4.2 2단계 증권 실시간 처리 관련 위협

- 외부 주식 시세 API Key가 노출될 수 있다.
- 시세 API 응답 형식이 바뀌거나 비정상 값이 들어올 수 있다.
- 모의 주문 생성기가 과도하게 실행되어 서버 부하를 만들 수 있다.
- 주문 가격, 수량에 비정상 값이 들어올 수 있다.
- 예수금보다 큰 매수 주문이 들어올 수 있다.
- 보유 수량보다 큰 매도 주문이 들어올 수 있다.
- WebSocket 연결 정보나 내부 이벤트 구조가 과도하게 노출될 수 있다.
- 주문 Queue나 OrderBook 초기화 기능이 잘못 호출될 수 있다.

## 5. 민감정보 처리

이 프로젝트에서 민감정보로 볼 수 있는 항목은 다음과 같다.

- DB 접속 정보
- DB 사용자명
- DB 비밀번호
- 외부 주식 시세 API Key
- 외부 주식 시세 API Secret
- 외부 API Access Token
- 외부 API WebSocket 승인 키
- 환경 변수 전체 값
- 내부 서버 주소
- 배포 환경 설정값
- 실제 사용자의 개인정보가 포함될 수 있는 값

현재 프로젝트는 실제 고객 개인정보를 다루지 않는 것을 원칙으로 한다.

더미 데이터와 모의 계좌 데이터는 실제 개인정보가 아니어야 한다.

민감정보 처리 원칙:

- 시크릿을 Git에 커밋하지 않는다.
- 시크릿을 코드에 하드코딩하지 않는다.
- 시크릿을 README, PPT, 이슈, 채팅에 작성하지 않는다.
- `.env`, 환경 변수, 시크릿 관리 도구를 사용한다.
- 로그에 DB 비밀번호, API Key, Token을 남기지 않는다.
- 프론트엔드 코드에 비공개 API Key를 넣지 않는다.
- 외부 API 키가 노출되었다고 의심되면 즉시 교체한다.

## 6. 인증과 인가 기준

이 프로젝트는 인증/인가 중심 프로젝트가 아니다.

따라서 1단계 MVP에서는 복잡한 로그인, JWT, 권한 체계를 구현하지 않는다.

다만 보안상 위험한 API는 공개 배포 시 보호해야 한다.

## 6.1 1단계에서 보호가 필요한 API

다음 API는 운영 또는 외부 공개 시 보호 대상이다.

```txt
POST /api/settlements/run
POST /api/dummy-data/generate
DELETE /api/settlements
POST /api/settlements/retry
POST /api/admin/index/apply
POST /api/admin/database/reset
```

현재 구현된 API 중 특히 `POST /api/settlements/run`은 배치 실행을 유발하므로 무제한 호출되면 안 된다.

보호 기준:

- 로컬 포트폴리오 실행에서는 인증 없이 사용 가능하다.
- 외부 공개 또는 배포 시에는 최소한 관리자 토큰, Basic Auth, IP 제한, 내부망 제한 중 하나를 적용한다.
- 위험한 API는 프론트 버튼만 숨기는 방식으로 보호하지 않는다.
- 서버에서 실행 가능 여부를 검증한다.

---

## 6.2 2단계에서 보호가 필요한 API

2단계에서 다음 API는 보호 대상이다.

```txt
POST /api/admin/order-generator/start
POST /api/admin/order-generator/stop
POST /api/admin/orderbook/reset
POST /api/admin/mock-price/start
POST /api/admin/mock-price/stop
POST /api/orders
```

특히 모의 주문 생성기 API는 초당 주문 수를 조절하므로 무제한 접근을 허용하면 서버 부하를 만들 수 있다.

보호 기준:

- 로컬 테스트에서는 제한된 환경에서만 실행한다.
- 외부 공개 시 관리자성 API는 반드시 보호한다.
- 초당 주문 수에는 상한선을 둔다.
- 주문 생성기 시작/중지는 로그로 남긴다.

---

## 7. API 보안

모든 API는 안전한 기본값을 가져야 한다.

## 7.1 1단계 배치 API 보안

정산 배치 API에서 검증할 값:

- 정산일자 형식
- 정산일자가 미래 날짜인지 여부
- 처리 전략 값
- 중복 실행 여부
- 조회 범위
- 페이지 크기
- 정렬 기준

원칙:

- 허용된 처리 전략만 받는다.
- 목록 조회에는 필요한 경우 페이지네이션을 적용한다.
- 정산 결과 전체를 무제한 반환하지 않는다.
- Entity를 그대로 반환하지 않고 DTO로 변환한다.
- 에러 응답에는 내부 스택 트레이스를 포함하지 않는다.

---

## 7.2 2단계 주문·체결 API 보안

주문 API에서 검증할 값:

- 종목 코드
- 주문 유형
- 주문 가격
- 주문 수량
- 계좌 ID
- 주문 가능 금액
- 보유 수량
- 허용된 주문 상태

원칙:

- 주문 가격은 0보다 커야 한다.
- 주문 수량은 0보다 커야 한다.
- 매수 주문은 예수금 부족 시 거절한다.
- 매도 주문은 보유 수량 부족 시 거절한다.
- 허용되지 않은 종목 코드는 거절한다.
- 주문 상태는 서버에서 결정한다.
- 클라이언트가 전달한 체결 상태나 잔고 변경값을 신뢰하지 않는다.

---

## 8. 입력값 검증

외부 입력값은 신뢰하지 않는다.

외부 입력값 예시:

- HTTP 요청 본문
- Query Parameter
- Path Parameter
- Header
- 외부 주식 시세 API 응답
- WebSocket 메시지
- 환경 변수
- 프론트엔드 입력값

검증 원칙:

- 타입, 길이, 형식, 허용 값을 검증한다.
- 예상하지 못한 필드는 거부하거나 무시한다.
- 날짜는 허용 가능한 범위인지 확인한다.
- 금액은 음수 또는 비정상 소수점 값을 거부한다.
- 수량은 양수 정수만 허용한다.
- 처리 전략은 Enum으로 제한한다.
- 주문 상태는 클라이언트 입력이 아니라 서버 로직으로 결정한다.
- SQL은 파라미터 바인딩 또는 JPA/ORM 방식을 사용한다.

1단계 입력 검증 예시:

```txt
settlementDate는 YYYY-MM-DD 형식이어야 한다.
processingStrategy는 BASIC_LOOP, GROUP_BY_QUERY, GROUP_BY_BULK_SAVE, GROUP_BY_BULK_INDEX 중 하나여야 한다.
pageSize는 상한선을 둔다.
```

현재 실행 가능한 정산 전략은 `BASIC_LOOP`, `GROUP_BY_QUERY`, `GROUP_BY_BULK_SAVE`이다. `GROUP_BY_BULK_INDEX`는 enum 값으로는 허용되지만 아직 구현되지 않았으므로 서버에서 명확한 미구현 오류로 응답해야 한다.

2단계 입력 검증 예시:

```txt
stockCode는 등록된 종목 코드여야 한다.
orderType은 BUY 또는 SELL이어야 한다.
orderPrice는 0보다 커야 한다.
orderQuantity는 0보다 큰 정수여야 한다.
ordersPerSecond는 허용된 상한선을 넘지 않아야 한다.
```

## 9. 시크릿 관리

시크릿은 소스 코드 밖에서 관리한다.

이 프로젝트에서 시크릿에 해당하는 값:

- DB 비밀번호
- 외부 주식 시세 API Key
- 외부 주식 시세 API Secret
- 외부 API Access Token
- WebSocket 승인 키
- 배포 환경 변수

시크릿 관리 원칙:

- 시크릿을 Git에 커밋하지 않는다.
- 애플리케이션 코드에 시크릿을 하드코딩하지 않는다.
- 프론트엔드 코드에 비공개 시크릿을 넣지 않는다.
- 환경 변수 또는 `.env` 파일을 사용한다.
- `.env` 파일은 Git에 포함하지 않는다.
- 문서에는 실제 시크릿 값을 적지 않는다.
- 예시 값은 `YOUR_API_KEY`, `YOUR_DB_PASSWORD`처럼 더미 값으로 작성한다.
- 노출이 의심되면 즉시 교체한다.

## 10. 로그와 에러 처리

로그는 문제 추적에 필요하지만, 민감정보를 노출할 수 있다.

## 10.1 로그에 남겨도 되는 정보

1단계 배치 로그:

```txt
정산일자
처리 전략
처리 건수
성공 건수
실패 건수
실행 시간
배치 상태
```

2단계 주문·체결 로그:

```txt
종목 코드
주문 ID
체결 ID
주문 상태
체결 수량
Queue 대기 건수
시세 출처 API/MOCK
```

---

## 10.2 로그에 남기면 안 되는 정보

```txt
DB 비밀번호
외부 API Key
외부 API Secret
Access Token
WebSocket 승인 키
Authorization Header 전체
환경 변수 전체 덤프
민감한 요청 본문 전체
```

---

## 10.3 에러 응답 원칙

에러 응답은 일관된 형식을 사용한다.

예시:

```json
{
  "code": "SETTLEMENT_DUPLICATED",
  "message": "이미 같은 조건의 정산 결과가 있습니다.",
  "detail": "settlementDate=2026-05-11, strategy=BASIC_LOOP",
  "timestamp": "2026-05-11T10:00:00"
}
```

주의할 점:

- 스택 트레이스를 사용자에게 반환하지 않는다.
- DB 연결 문자열을 반환하지 않는다.
- API Key 또는 시크릿 값을 반환하지 않는다.
- 내부 패키지 구조를 과도하게 노출하지 않는다.
- 사용자가 다음 행동을 알 수 있는 메시지를 제공한다.

---

## 11. 데이터 저장 보안

이 프로젝트는 실제 고객 개인정보를 저장하지 않는다.

더미 데이터와 모의 데이터는 실제 개인정보가 아니어야 한다.

저장 데이터 원칙:

- 금액 데이터는 `BigDecimal` 기준으로 저장한다.
- 정산 결과는 중복 저장되지 않도록 제약조건을 검토한다.
- 원천 Payment 데이터와 Settlement 결과는 대사 가능해야 한다.
- BatchJobHistory는 성공/실패 여부와 관계없이 보존한다.
- 외부 API Key와 Secret은 DB에 저장하지 않는다.
- 모의 계좌번호는 실제 계좌번호 형식을 그대로 사용하지 않는다.
- 모의 주문과 체결 데이터는 실제 투자 데이터가 아님을 문서에 명시한다.

## 12. 외부 주식 시세 API 보안

2단계에서는 실제 또는 모의 주식 시세 데이터를 사용할 수 있다.

외부 주식 시세 API를 사용할 경우 다음 기준을 지킨다.

- API Key와 Secret은 서버 환경 변수로 관리한다.
- 프론트엔드에서 외부 API를 직접 호출하지 않는다.
- 서버가 외부 API를 호출하고 필요한 데이터만 프론트로 전달한다.
- API 응답값을 검증한다.
- 비정상 가격, 음수 가격, 누락된 종목 코드는 거부한다.
- API 실패 시 모의 시세로 대체할 수 있어야 한다.
- 호출 제한을 고려한다.
- API 실패 로그에는 API Key를 남기지 않는다.

권장 흐름:

```txt
Server
→ External Stock Price API
→ Validate Response
→ Store or Cache Current Price
→ Send Safe Price DTO to Frontend
```

외부 API 실패 시 흐름:

```txt
시세 API 호출 실패
→ 실패 로그 기록
→ 마지막 정상 현재가 사용
→ 마지막 현재가가 없으면 모의 시세 생성
→ priceSource = MOCK 표시
```

## 13. 프론트엔드 보안

프론트엔드는 단독 보안 계층으로 신뢰하지 않는다.

프론트엔드 보안 원칙:

- 프론트엔드 코드에 API Key나 DB 정보를 넣지 않는다.
- 숨겨진 버튼이나 비활성화된 UI만으로 위험한 API를 보호하지 않는다.
- 서버에서 입력값을 다시 검증한다.
- 화면에는 필요한 데이터만 표시한다.
- 에러 메시지에 내부 구조를 노출하지 않는다.
- WebSocket 메시지도 서버에서 검증한 결과만 전송한다.

1단계 프론트 주의사항:

- 정산 결과를 무제한으로 렌더링하지 않는다.
- 배치 실행 버튼 중복 클릭을 방지한다.
- 실패 메시지는 사용자에게 필요한 수준으로만 보여준다.

2단계 프론트 주의사항:

- API Key를 프론트에 넣지 않는다.
- 주문 생성기 실행 조건을 명확히 표시한다.
- 주문 가격과 수량 입력값을 프론트에서도 1차 검증하되, 서버 검증을 반드시 유지한다.
- WebSocket으로 받은 데이터는 화면 표시 용도로만 사용하고, 잔고 계산 기준으로 삼지 않는다.
