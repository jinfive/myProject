
# Design

## 1. 목적

이 문서는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 제품과 인터페이스 설계 기준을 정의한다.

이 프로젝트의 화면은 단순히 기능을 실행하기 위한 UI가 아니다.

포트폴리오, README, PPT, 면접 설명에서 다음 흐름을 보여주기 위한 도구이다.

```txt
어떤 문제가 있었는가
→ 그 문제를 왜 중요하게 봤는가
→ 해결을 위해 어떤 생각을 했는가
→ 어떤 액션을 했는가
→ 어떻게 검증했는가
→ 결과를 어떻게 설명할 것인가
````

따라서 화면 설계의 핵심은 “기능 나열”이 아니라, **문제를 발견하고 개선해 나간 과정을 사용자가 한눈에 이해할 수 있게 만드는 것**이다.

이 문서는 다음 내용을 다룬다.

* 제품 설계 원칙
* 포트폴리오 중심 화면 설계 기준
* 1단계 배치 정산 대시보드 설계 기준
* 2단계 증권 실시간 주문·체결 화면 설계 기준
* 상태, 에러, 빈 화면 처리 기준
* 성능 지표 표시 기준
* 정합성 검증 결과 표시 기준
* PPT와 자소서 설명을 위한 화면 구성 기준
* 에이전트 작업 지침

---

## 2. 프로젝트 화면의 역할

이 프로젝트의 화면은 두 가지 역할을 가진다.

### 2.1 기능 실행 화면

사용자가 정산 배치를 실행하거나, 실시간 주문·체결 흐름을 확인할 수 있어야 한다.

예시:

* 정산일자 선택
* 처리 전략 선택
* 정산 배치 실행
* 정산 결과 조회
* 배치 실행 이력 조회
* 성능 비교 확인
* 실시간 시세 확인
* 모의 주문 생성
* 체결 결과 확인
* 잔고와 예수금 변경 확인

---

### 2.2 포트폴리오 설명 화면

프로젝트를 제출하거나 면접에서 설명할 때, 화면만 보고도 다음 내용이 드러나야 한다.

* 어떤 문제가 있었는가
* 어떤 방식으로 처음 구현했는가
* 처음 방식의 한계는 무엇인가
* 어떤 개선 전략을 적용했는가
* 개선 전후 성능 차이는 어떤가
* 데이터 정합성은 어떻게 확인했는가
* 실패나 재실행 상황은 어떻게 관리했는가
* 2단계 증권 실시간 처리로 어떻게 확장할 수 있는가

즉, 화면은 단순 결과표가 아니라 **개선 과정의 증거**가 되어야 한다.

---

## 3. 전체 프로젝트 단계

이 프로젝트는 하나의 저장소 안에서 두 단계로 진행한다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

현재는 1단계인 **정산 배치 성능 비교 프로젝트**를 먼저 완성한다.

2단계는 1단계 완료 후, 실시간 주식 시세 API 또는 모의 시세 데이터를 기반으로 주문을 발생시키고, Queue, OrderBook, Matching Engine, WebSocket을 활용해 실시간 주문·체결 흐름을 구현한다.

---

## 4. 핵심 설계 키워드

이 프로젝트의 제품과 화면은 다음 키워드를 중심으로 설계한다.

| 키워드    | 화면에서 보여줘야 할 내용                     |
| ------ | ---------------------------------- |
| 안정성    | 배치 실행 이력, 실패 이력, 재실행 가능 여부         |
| 정합성    | 원천 데이터와 정산 결과 대사, 체결 결과와 잔고/예수금 일치 |
| 성능 개선  | 처리 전략별 실행 시간, 개선율, 병목 원인           |
| 실시간성   | 시세 수신, 주문 발생, 체결 알림, 처리 지연시간       |
| 액션 중심  | 문제를 어떻게 판단했고 어떤 구현을 했는지            |
| 설명 가능성 | README, PPT, 자소서에서 바로 설명 가능한 화면 구성 |

---

## 5. 액션 중심 설계 원칙

이 프로젝트는 결과 중심이 아니라 **액션 중심**으로 설명되어야 한다.

따라서 화면과 문서는 다음 흐름을 따른다.

```txt
문제 상황
→ 판단 기준
→ 선택한 해결 방식
→ 실제 구현 액션
→ 검증 방법
→ 결과
→ 배운 점
→ 포트폴리오/면접 설명 포인트
```

예를 들어 성능 개선 화면은 단순히 다음처럼 끝나면 안 된다.

```txt
BASIC_LOOP: 10초
GROUP_BY_QUERY: 3초
```

대신 다음 정보가 함께 보여야 한다.

```txt
문제:
BASIC_LOOP는 Payment 전체 데이터를 조회해 Java 반복문으로 집계했다.

판단:
데이터가 증가할수록 전체 Entity 로딩과 반복 집계가 병목이 될 수 있다고 봤다.

액션:
DB GROUP BY 집계를 적용해 필요한 집계 결과만 조회하도록 바꿨다.

검증:
동일한 정산일자와 동일한 데이터 건수로 처리 시간을 비교했다.

결과:
처리 시간이 감소했고, 정산 결과는 대사 기능으로 검증했다.
```

---

## 6. 기본 설계 원칙

제품과 인터페이스는 다음 원칙을 따른다.

* 사용자가 현재 화면의 목적을 바로 이해할 수 있어야 한다.
* 중요한 지표와 행동이 먼저 보여야 한다.
* 성능 개선 전후 차이를 한눈에 비교할 수 있어야 한다.
* 정산 결과와 대사 결과를 함께 확인할 수 있어야 한다.
* 배치 실행 상태와 실패 원인을 추적할 수 있어야 한다.
* 실시간 처리 화면에서는 현재가, 주문, 체결, 잔고 변경이 연결되어 보여야 한다.
* 내부 개발자 용어만 나열하지 않고, 금융 업무 흐름으로 설명한다.
* README와 PPT에 그대로 옮겨도 설명 가능한 구조로 만든다.
* 기능보다 사용자의 이해와 설명 가능성을 우선한다.

## 7. 1단계 배치 정산 화면 설계

1단계 화면은 **대용량 금융 거래 정산·대사 배치 성능 최적화**를 설명하기 위한 화면이다.

핵심 목표는 다음과 같다.

```txt
대용량 거래 데이터를 어떻게 정산했는가
처음 방식의 병목은 무엇이었는가
어떤 개선 전략을 적용했는가
성능은 어떻게 달라졌는가
정산 결과는 어떻게 검증했는가
```

---

## 8. 1단계 필수 화면

### 8.1 정산 실행 대시보드

정산 배치를 실행하는 메인 화면이다.

표시 항목:

* 정산일자 선택
* 처리 전략 선택

    * BASIC_LOOP
    * GROUP_BY_QUERY
    * GROUP_BY_BULK_SAVE
    * GROUP_BY_BULK_INDEX
* 정산 배치 실행 버튼
* 실행 상태
* 실행 시간
* 처리 건수
* 성공 건수
* 실패 건수

설계 기준:

* 사용자는 어떤 전략을 실행하는지 명확히 알아야 한다.
* 실행 전 선택한 정산일자와 전략을 확인할 수 있어야 한다.
* 실행 중에는 중복 클릭을 막아야 한다.
* 실행 완료 후 결과 화면으로 자연스럽게 연결되어야 한다.

버튼 문구 예시:

```txt
정산 배치 실행
BASIC_LOOP 실행
GROUP_BY_QUERY 실행
성능 비교 실행
```

---

### 8.2 성능 비교 화면

이 프로젝트의 핵심 화면이다.

표시 항목:

* 처리 전략
* 데이터 건수
* 정산일자
* 처리 시간
* 개선 내용
* 개선율
* 병목 원인
* 적용한 액션

표 예시:

| 전략             |   데이터 건수 | 처리 시간 | 병목/개선 내용            | 액션              |
| -------------- | -------: | ----: | ------------------- | --------------- |
| BASIC_LOOP     | 100,000건 |   측정값 | 전체 조회 후 Java 반복문 집계 | 기준선 구현          |
| GROUP_BY_QUERY | 100,000건 |   측정값 | DB GROUP BY 집계      | 전체 Entity 로딩 제거 |
| GROUP_BY_BULK_SAVE | 100,000건 |   측정값 | DB GROUP BY 집계 + 정산 결과 일괄 저장 | 반복 save 감소 |
| GROUP_BY_BULK_INDEX | 100,000건 |   측정값 | DB GROUP BY 집계 + 일괄 저장 + 조회 조건 인덱스 적용 | 정산일자/가맹점 조회 최적화 |

설계 기준:

* 결과 수치만 보여주지 않는다.
* “왜 이 전략을 적용했는지”가 함께 보여야 한다.
* 개선 전후를 같은 조건으로 비교했음을 명확히 표시한다.
* README와 PPT에 그대로 사용할 수 있는 표 형태를 유지한다.

---

### 8.3 정산 결과 화면

가맹점별 정산 결과를 보여주는 화면이다.

표시 항목:

* 가맹점명
* 총 결제금액
* 총 취소금액
* 순매출
* 수수료
* 최종 정산금액
* 처리 전략
* 정산일자

설계 기준:

* 금액은 오른쪽 정렬한다.
* 금액 단위는 일관되게 표시한다.
* 결제금액, 취소금액, 정산금액의 관계가 이해되어야 한다.
* 처리 전략별 결과를 비교할 수 있어야 한다.

---

### 8.4 대사 결과 화면

정산 결과와 원천 Payment 데이터를 비교하는 화면이다.

표시 항목:

* 정산일자
* 처리 전략
* Payment 기준 총 결제금액
* Settlement 기준 총 결제금액
* Payment 기준 총 취소금액
* Settlement 기준 총 취소금액
* 일치 여부
* 불일치 원인

설계 기준:

* 정합성 검증 결과가 명확해야 한다.
* 일치/불일치 상태는 텍스트와 색상 또는 배지를 함께 사용한다.
* 색상만으로 상태를 전달하지 않는다.
* 불일치 발생 시 다음 행동을 안내한다.

상태 문구 예시:

```txt
대사 일치
대사 불일치
원천 데이터와 정산 결과가 일치하지 않습니다.
불일치 원인을 확인한 뒤 재실행 여부를 판단하세요.
```

---

### 8.5 배치 실행 이력 화면

배치 실행 기록을 보여주는 화면이다.

표시 항목:

* 실행일시
* 정산일자
* 처리 전략
* 실행 상태
* 처리 건수
* 성공 건수
* 실패 건수
* 실행 시간
* 실패 원인

설계 기준:

* 최신 실행 이력이 먼저 보여야 한다.
* 전략별 필터가 가능하면 좋다.
* 실패 상태는 원인을 확인할 수 있어야 한다.
* 성능 비교 화면과 연결되어야 한다.

---

### 8.6 실패 이력 화면

배치 실패나 가맹점별 정산 실패를 보여주는 화면이다.

표시 항목:

* 실패 발생 시간
* 정산일자
* 처리 전략
* 가맹점
* 실패 코드
* 실패 메시지
* 재처리 가능 여부

설계 기준:

* 실패는 숨기지 않는다.
* 실패 원인과 다음 행동을 안내한다.
* 재처리 가능 여부를 명확히 표시한다.
* 안정성 관점에서 포트폴리오 설명에 활용할 수 있어야 한다.

---

## 9. 2단계 증권 실시간 처리 화면 설계

2단계는 **실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템**이다.

핵심 목표는 다음 흐름을 보여주는 것이다.

```txt
실시간 주식 시세 API 수신
→ 현재가 기반 모의 주문 생성
→ 주문 Queue 적재
→ 메모리 기반 OrderBook 처리
→ 가격 우선·시간 우선 체결
→ 주문 상태 변경
→ 잔고와 예수금 반영
→ 체결 결과 DB 저장
→ WebSocket 실시간 알림
→ 초당 주문 수 기준 성능 테스트
```

2단계 화면은 증권사 포트폴리오에서 다음 역량을 보여주기 위한 화면이다.

* 실시간 처리
* 주문·체결 흐름 이해
* 동시성 고려
* Queue 기반 처리
* OrderBook 기반 매칭
* 잔고/예수금 정합성
* WebSocket 알림
* 대량 주문 성능 테스트

---

## 10. 2단계 필수 화면

### 10.1 실시간 시세 화면

실제 주식 시세 API 또는 모의 시세 생성기를 통해 받은 현재가를 표시한다.

표시 항목:

* 종목명
* 종목 코드
* 현재가
* 등락률
* 거래량
* 시세 출처

    * API
    * MOCK
* 마지막 수신 시간

설계 기준:

* 시세 출처가 API인지 MOCK인지 명확히 표시한다.
* 외부 API 실패 시 대체 흐름을 표시할 수 있어야 한다.
* 현재가는 주문 생성 기준으로 사용된다는 점을 설명할 수 있어야 한다.

상태 문구 예시:

```txt
실시간 시세 수신 중
모의 시세 사용 중
시세 API 연결 실패로 모의 시세를 사용합니다.
```

---

### 10.2 모의 주문 생성기 화면

실제 사용자가 없어도 대량 주문 상황을 만들기 위한 화면이다.

표시 항목:

* 주문 생성 시작/중지 버튼
* 초당 주문 수
* 매수/매도 비율
* 가격 범위

    * 현재가 기준 ±1%
    * 현재가 기준 ±3%
* 생성된 주문 수
* Queue 대기 건수

설계 기준:

* 사용자가 부하 테스트 조건을 이해할 수 있어야 한다.
* 주문 생성 조건이 명확히 표시되어야 한다.
* 시작 후 중지할 수 있어야 한다.
* 생성된 주문이 Queue와 체결 결과로 이어지는 흐름이 보여야 한다.

버튼 문구 예시:

```txt
주문 생성 시작
주문 생성 중지
초당 100건으로 실행
```

* 매수/매도 선택
* 주문 가격
* 주문 수량

설계 기준:

* 매수 주문 시 예수금 부족 여부를 안내한다.
* 매도 주문 시 보유 수량 부족 여부를 안내한다.
* 주문 접수 후 주문 상태가 표시되어야 한다.
* 거절된 주문은 거절 사유가 표시되어야 한다.

## 12. 로딩 상태

로딩 상태는 사용자가 기다려야 하는 상황을 알려준다.

원칙:

* 데이터가 늦게 불러와질 수 있는 화면에는 로딩 상태를 제공한다.
* 배치 실행처럼 시간이 걸리는 작업은 진행 중임을 명확히 보여준다.

* 사용자가 중복 클릭하지 않도록 처리한다.
* 로딩 중에도 가능한 경우 화면 구조를 유지한다.

예시:

```txt
정산 배치를 실행 중입니다.
성능 측정 결과를 불러오는 중입니다.
실시간 시세를 연결하는 중입니다.
주문 Queue를 처리하는 중입니다.
```

## 13. 에러 상태

에러 상태는 원인과 다음 행동을 안내해야 한다.

원칙:

* 에러 메시지는 사용자 입장에서 이해할 수 있어야 한다.
* 내부 시스템 용어를 그대로 노출하지 않는다.
* 사용자가 할 수 있는 다음 행동을 안내한다.
* 재시도가 가능한 경우 재시도 버튼을 제공한다.
* 보안상 민감한 정보는 에러 메시지에 포함하지 않는다.
* 포트폴리오 설명에 필요한 실패 원인은 별도 실패 이력에 남긴다.

1단계 에러 예시:

```txt
정산 배치를 실행하지 못했습니다. 정산일자와 DB 연결 상태를 확인해 주세요.
이미 같은 조건의 정산 결과가 있습니다. 재실행 정책을 확인해 주세요.
정산 결과와 원천 데이터가 일치하지 않습니다. 대사 결과를 확인해 주세요.
```

2단계 에러 예시:

```txt
시세 API 연결에 실패했습니다. 모의 시세로 전환합니다.
예수금이 부족해 매수 주문을 접수할 수 없습니다.
보유 수량이 부족해 매도 주문을 접수할 수 없습니다.
WebSocket 연결이 끊겼습니다. 다시 연결을 시도합니다.
```

## 16. 위험한 행동

다음 작업은 되돌리기 어렵거나 데이터 정합성에 영향을 줄 수 있으므로 신중하게 설계한다.

1단계 위험한 행동:

* 정산 결과 삭제
* 같은 정산일자 재실행
* 더미 데이터 초기화
* 인덱스 변경
* DB 초기화

2단계 위험한 행동:

* 모의 계좌 초기화
* 보유 수량 초기화
* 예수금 초기화
* 주문 Queue 초기화
* OrderBook 초기화
* 주문 생성기 대량 실행

원칙:

* 사용자가 행동의 결과를 이해할 수 있어야 한다.
* 필요한 경우 확인 단계를 둔다.
* 되돌릴 수 없는 작업은 명확히 안내한다.
* 가능하면 복구 방법을 제공한다.

예시:

```txt
이 작업은 기존 정산 결과를 삭제합니다.
정말 재실행하시겠습니까?
```

```txt
주문 Queue와 OrderBook이 초기화됩니다.
진행 중인 모의 주문이 모두 사라질 수 있습니다.
```

## 17. 문구 작성 기준

인터페이스 문구는 짧고 명확해야 한다.

원칙:

* 사용자가 이해하기 쉬운 표현을 사용한다.
* 내부 용어나 개발자 중심 표현을 피한다.
* 하지만 포트폴리오 설명에 필요한 기술 용어는 함께 표시할 수 있다.
* 한 문장에 하나의 의미만 담는다.
* 에러 문구는 비난하지 않고 해결 방법을 안내한다.
* 액션 중심 문구를 사용한다.

좋은 문구:

```txt
전체 데이터를 조회해 Java 반복문으로 집계했습니다.
DB GROUP BY로 필요한 집계 결과만 조회하도록 개선했습니다.
반복 저장을 줄이기 위해 일괄 저장을 적용했습니다.
정산 결과와 원천 데이터를 대사했습니다.
```

피해야 할 문구:

```txt
성능이 좋아졌습니다.
최적화했습니다.
처리했습니다.
완료했습니다.
```

이유 없이 결과만 말하는 문구는 피한다.

## 18. 일관성

제품 안의 표현과 동작은 일관되어야 한다.

확인할 것:

* 같은 기능은 같은 이름으로 부르는가
* 같은 처리 전략은 같은 이름으로 표시되는가
* 같은 상태는 같은 배지와 문구로 표현되는가
* 같은 행동은 같은 위치와 방식으로 제공되는가
* 버튼 문구와 알림 문구가 일관되는가
* 1단계와 2단계 성능 비교 표 구조가 일관되는가

용어 기준:

| 용어                    | 사용 기준                      |
| --------------------- | -------------------------- |
| BASIC_LOOP            | 전체 조회 후 Java 반복문 집계        |
| GROUP_BY_QUERY        | DB GROUP BY 기반 집계          |
| GROUP_BY_BULK_SAVE    | DB GROUP BY 집계 후 정산 결과 일괄 저장 |
| GROUP_BY_BULK_INDEX   | DB GROUP BY 집계, 일괄 저장, 인덱스 적용 후 처리 |
| OrderBook             | 종목별 매수/매도 주문 장부            |
| Matching Engine       | 주문을 가격 우선·시간 우선으로 체결하는 처리기 |
| VirtualOrderGenerator | 현재가 기준으로 모의 주문을 생성하는 기능    |

## 21. 디자인 시스템

현재는 복잡한 디자인 시스템보다 **명확한 정보 전달**을 우선한다.

우선 적용할 구성:

* 카드형 지표 영역
* 전략별 비교 표
* 실행 이력 테이블
* 상태 배지
* 에러/성공 메시지
* 간단한 필터
* 정산일자 선택
* 처리 전략 선택

원칙:

* 새로운 스타일을 계속 추가하지 않는다.
* 기존 화면 패턴을 재사용한다.
* 화려한 디자인보다 설명 가능성을 우선한다.
* PPT에 캡처했을 때 읽기 쉬운 구조를 만든다.

## 22. 제품 리뷰 체크리스트

기능을 설계하거나 수정하기 전에 다음을 확인한다.

* [ ] 이 화면이 해결하는 문제가 명확한가?
* [ ] 사용자가 첫 화면에서 무엇을 해야 하는지 알 수 있는가?
* [ ] 주요 행동이 명확하게 보이는가?
* [ ] 문제 → 액션 → 검증 → 결과 흐름이 드러나는가?
* [ ] 성능 비교 수치가 이해하기 쉽게 표시되는가?
* [ ] 정합성 검증 결과가 보이는가?
* [ ] 실패했을 때 원인과 다음 행동이 안내되는가?
* [ ] 빈 상태가 준비되어 있는가?
* [ ] 로딩 상태가 준비되어 있는가?
* [ ] 위험한 행동은 충분히 확인하는가?
* [ ] 문구가 사용자 입장에서 이해하기 쉬운가?
* [ ] README와 PPT에 그대로 사용할 수 있는 화면인가?
* [ ] 기존 화면과 일관성이 있는가?
* [ ] 1단계 완료 전 2단계 화면을 과하게 구현하고 있지 않은가?

2단계 증권 실시간 처리 화면을 설계할 때는 추가로 확인한다.

* [ ] 시세 출처가 API인지 MOCK인지 표시되는가?
* [ ] 현재가 기반 주문 생성 흐름이 보이는가?
* [ ] 주문 Queue 대기 건수가 보이는가?
* [ ] OrderBook에서 가격 우선·시간 우선 원칙이 이해되는가?
* [ ] 체결 결과가 주문 상태와 연결되는가?
* [ ] 체결 결과가 잔고와 예수금 변경으로 이어지는가?
* [ ] WebSocket 알림이 실시간성을 보여주는가?
* [ ] 초당 주문 수 기준 성능 테스트 결과가 표시되는가?

## 23. 에이전트 작업 지침

AI 에이전트가 제품이나 인터페이스를 설계할 때는 다음을 따른다.

* 이 문서를 먼저 확인한다.
* 현재 프로젝트 단계가 1단계인지 2단계인지 확인한다.
* 현재는 1단계 배치 성능 개선 화면을 우선한다.
* 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 설계한다.
* 기존 화면과 문구 패턴을 우선한다.
* 새로운 UI 패턴을 불필요하게 만들지 않는다.
* 화면에는 하나의 핵심 목적을 둔다.
* 기능 나열보다 문제 해결 과정을 보여준다.
* 로딩, 에러, 빈 상태를 함께 고려한다.
* 위험한 행동에는 확인 절차를 둔다.
* 내부 용어만 쓰지 말고 사용자가 이해할 수 있는 설명을 붙인다.
* 성능 지표와 정합성 지표를 함께 보여준다.
* PPT 캡처와 자소서 설명에 사용할 수 있는 화면인지 확인한다.
* 설계가 커질 경우 별도 design-doc을 작성한다.

## 24. 핵심 원칙 요약

* 기능보다 문제 해결 과정을 먼저 보여준다.
* 결과 중심이 아니라 액션 중심으로 설계한다.
* 화면은 한 번에 이해할 수 있어야 한다.
* 중요한 행동과 지표는 명확하게 보여야 한다.
* 성능 비교는 병목과 개선 액션을 함께 보여준다.
* 정합성 검증 결과는 화면에 드러나야 한다.
* 실패와 빈 상태도 설계의 일부다.
* 문구는 짧고 구체적으로 작성한다.
* 기존 패턴을 유지한다.
* 사용자의 실수를 줄이는 방향으로 설계한다.
* 1단계는 배치 정산 성능 개선을 보여준다.
* 2단계는 실시간 주식 시세 API 기반 주문·체결 처리를 보여준다.
* README, PPT, 자소서에서 설명 가능한 화면을 만든다.
