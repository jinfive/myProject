# Product Specs

## 1. 목적

이 디렉터리는 제품 기능별 요구사항과 정책을 관리한다.

제품 스펙은 다음 내용을 명확히 하기 위해 작성한다.

- 어떤 기능을 만들 것인가
- 누가 사용하는 기능인가
- 사용자는 어떤 문제를 해결하려 하는가
- 기능의 정상 흐름은 무엇인가
- 예외 상황은 무엇인가
- 권한과 접근 조건은 무엇인가
- 완료 기준은 무엇인가

---

## 2. 문서 역할

제품 스펙 문서는 기능 구현 전에 기준이 되는 문서이다.

개발자와 에이전트는 기능을 구현하거나 수정하기 전에 관련 제품 스펙을 먼저 확인한다.

제품 스펙은 다음 작업에 사용된다.

- 기능 요구사항 확인
- API 설계 기준 확인
- 화면 흐름 확인
- 권한 조건 확인
- 예외 상황 확인
- 테스트 케이스 작성
- 구현 범위 조정

## 5. 요구사항 작성 원칙

요구사항은 명확하고 검증 가능해야 한다.

좋은 요구사항:

```txt
사용자는 이메일과 비밀번호로 로그인할 수 있다.
로그인 실패 시 사용자에게 실패 메시지를 보여준다.
관리자는 다른 사용자의 상태를 변경할 수 있다.
```

좋지 않은 요구사항:

```txt
로그인을 잘 만든다.
관리 기능을 추가한다.
사용하기 편하게 만든다.
```

요구사항은 구현자가 무엇을 만들어야 하는지 바로 이해할 수 있어야 한다.

## 6. 권한 작성 기준

권한이 필요한 기능은 반드시 접근 조건을 적는다.

예시:

```md
## 권한

- 로그인한 사용자만 접근할 수 있다.
- 일반 사용자는 본인 데이터만 조회할 수 있다.
- 관리자는 전체 데이터를 조회할 수 있다.
- 비활성화된 계정은 접근할 수 없다.
```

권한 조건이 불명확하면 구현 전에 먼저 정리한다.

---

## 7. 예외 상황 작성 기준

제품 스펙에는 정상 흐름뿐 아니라 예외 상황도 포함한다.

예외 상황 예시:

* 입력값이 잘못된 경우
* 권한이 없는 경우
* 데이터가 없는 경우
* 이미 존재하는 데이터인 경우
* 외부 서비스가 실패한 경우
* 요청이 중복된 경우
* 사용자가 비활성 상태인 경우

예외 상황은 API, 화면, 테스트 설계에 직접 영향을 준다.

---

## 8. 완료 기준

각 기능은 완료 기준을 가져야 한다.

완료 기준 예시:

```md
## 완료 기준

- [ ] 사용자가 정상적으로 기능을 사용할 수 있다.
- [ ] 잘못된 입력에 대한 처리가 있다.
- [ ] 권한 없는 사용자는 접근할 수 없다.
- [ ] 필요한 API가 구현되어 있다.
- [ ] 필요한 화면 상태가 정의되어 있다.
- [ ] 테스트 또는 수동 검증이 가능하다.
- [ ] 관련 문서가 갱신되어 있다.
```

완료 기준이 없으면 작업이 끝났는지 판단하기 어렵다.

---

## 9. 에이전트 작업 지침

AI 에이전트가 기능을 구현하거나 수정할 때는 다음을 따른다.

* 관련 제품 스펙을 먼저 확인한다.
* 제품 스펙이 없으면 필요한 범위만 간단히 작성하거나 사용자에게 확인한다.
* 요구사항을 임의로 확장하지 않는다.
* 권한 조건을 반드시 확인한다.
* 예외 상황을 함께 고려한다.
* 기능 변경 시 제품 스펙도 함께 갱신한다.
* 제품 스펙과 코드가 다르면 현재 동작과 변경 의도를 확인한 뒤 문서를 갱신한다.

---

## 10. 핵심 원칙 요약

* 제품 스펙은 기능의 기준이다.
* 구현 전에 사용자, 요구사항, 권한, 예외 상황을 확인한다.
* 기능은 명확한 완료 기준을 가져야 한다.
* 불명확한 요구사항은 작게 정리한 뒤 구현한다.
* 제품 스펙은 코드 변경에 맞춰 계속 갱신한다.
