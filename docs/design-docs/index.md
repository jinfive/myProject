# Design Docs

## 1. 목적

이 디렉터리는 **금융 거래 처리 성능 최적화 포트폴리오 프로젝트**의 상세 설계 문서를 관리한다.

이 프로젝트는 단순 기능 구현 프로젝트가 아니라, 금융 IT에서 중요한 다음 키워드를 코드와 문서로 증명하는 것을 목표로 한다.

- 안정성
- 데이터 정합성
- 대용량 배치 처리
- 성능 개선
- 실시간 시세 처리
- 주문·체결 처리
- 잔고와 예수금 정합성
- 운영 관점
- 포트폴리오 설명 가능성

프로젝트는 하나의 저장소 안에서 단계적으로 확장한다.

```txt
1단계: 대용량 금융 거래 정산·대사 배치 성능 최적화
2단계: 실시간 주식 시세 API 기반 모의 주문·체결 처리 시스템
```

현재는 1단계인 **정산 배치 성능 비교 프로젝트**를 먼저 완성한다.

2단계는 1단계 완료 후, 실제 주식 시세 API 또는 모의 시세 데이터를 기반으로 주문을 발생시키고, Queue, OrderBook, Matching Engine, WebSocket을 활용해 실시간 주문·체결 흐름을 구현하는 증권사 특화 확장 프로젝트로 진행한다.

이 디렉터리의 설계 문서는 기능을 구현하기 전에 다음 내용을 정리하기 위해 사용한다.

- 구조
- 흐름
- 책임
- 데이터
- 예외 상황
- 성능 병목
- 안정성 고려사항
- 정합성 검증 방식
- 대안 비교
- 포트폴리오 설명 포인트

## 2. 설계 문서의 역할

설계 문서는 코드 작성 전에 방향을 정리하는 문서다.

단순 작업 지시서가 아니라, 다음 질문에 답하기 위해 작성한다.

```txt
왜 이 구조를 선택했는가?
어떤 문제가 있었는가?
어떤 대안을 검토했는가?
어떤 기준으로 성능과 정합성을 검증할 것인가?
README와 PPT에서 어떻게 설명할 수 있는가?
```

특히 이 프로젝트에서는 설계 문서가 포트폴리오 자료의 기반이 된다.

따라서 설계 문서는 다음 내용을 명확히 남겨야 한다.

- 문제 정의
- 기존 방식의 한계
- 개선 구조
- 성능 측정 기준
- 데이터 정합성 보장 방식
- 실패 상황 대응 방식
- 향후 확장 가능성
- 자기소개서와 면접에서 설명할 수 있는 포인트

1단계 설계 문서는 다음 질문에 답해야 한다.

```txt
대용량 금융 거래 데이터를 배치로 정산할 때 어떤 병목이 있었고,
GROUP BY, 벌크 저장, 인덱스 적용으로 어떻게 개선했는가?
정산 결과의 정합성과 배치 안정성은 어떻게 관리했는가?
```

2단계 설계 문서는 다음 질문에 답해야 한다.

```txt
실시간 주식 시세 데이터를 어떻게 수신하고,
현재가를 기준으로 모의 주문을 어떻게 생성하며,
많은 주문을 Queue와 OrderBook으로 어떻게 처리하고,
체결 결과에 따른 잔고와 예수금 정합성을 어떻게 보장할 것인가?
```

## 3. 현재 프로젝트 단계

현재 프로젝트는 다음 상태를 기준으로 한다.

```txt
정산 배치 성능 비교용 MVP 중 BASIC_LOOP 기준선 구현 완료
```

현재 구현된 기능:

- Spring Boot 백엔드 구조 구성
- Merchant, Payment, Settlement, BatchJobHistory 도메인 구현
- 더미 데이터 자동 생성
    - 가맹점 100개
    - 결제 데이터 100,000건
    - 특정일 거래 약 70,000건
- BASIC_LOOP 정산 배치 구현
- 정산 실행 API 구현
- 정산 결과 조회 API 구현
- 배치 이력 조회 API 구현
- React 대시보드 구현
- 프론트 빌드 성공
- 백엔드 테스트 명령 성공

현재 API:

```txt
POST /api/settlements/run
GET  /api/settlements
GET  /api/batch-histories
```

현재 부족한 부분:

- GROUP BY 기반 개선 전략 미구현
- 벌크 저장 최적화 미구현
- 인덱스 적용 및 성능 비교 미구현
- 테스트 부족
- DB 설정 불일치
- 실제 테이블 기준 DB 문서 갱신 필요
- 실패 이력 저장 부족
- 재실행 정책 부족
- 대사 기능 미구현

2단계 증권 실시간 처리 프로젝트는 아직 구현하지 않는다.

다만 기획은 다음 방향으로 확정한다.

```txt
실시간 주식 시세 API 수신
→ 현재가 기반 모의 주문 생성
→ 주문 Queue 적재
→ 메모리 기반 OrderBook 처리
→ 가격 우선·시간 우선 체결
→ 주문 상태 변경
→ 잔고와 예수금 반영
→ 체결 결과 DB 저장
→ WebSocket 실시간 알림
→ 초당 주문 수 기준 성능 테스트
```

## 5. 현재 우선 설계 대상

현재 프로젝트에서 가장 먼저 설계해야 할 대상은 다음과 같다.

### 5.1 DB 설정 정리 설계

현재 `application.yml`과 `compose.yaml`의 DB 설정이 일치하지 않는다.

이 문제는 실행 안정성에 영향을 준다.

설계 문서에서 정리할 내용:

- 사용할 DB 이름
- 사용할 사용자명
- 사용할 비밀번호
- 로컬 실행 기준
- 테스트 실행 기준
- Docker Compose 실행 기준
- 문서와 실제 설정의 일치 기준

---

### 5.2 정산 계산 테스트 설계

현재 테스트가 거의 없으므로, 정산 결과의 신뢰도를 보여주기 어렵다.

설계 문서에서 정리할 내용:

- 결제금액 계산 기준
- 취소금액 차감 기준
- 순매출 계산 기준
- 수수료 계산 기준
- 최종 정산금액 계산 기준
- 반올림 기준
- 중복 정산 방지 기준

---

### 5.3 GROUP BY 개선 전략 설계

현재는 BASIC_LOOP만 구현되어 있다.

포트폴리오 핵심은 성능 개선 비교이므로 GROUP BY 개선 전략이 반드시 필요하다.

설계 문서에서 정리할 내용:

- BASIC_LOOP의 병목
- GROUP BY 쿼리 구조
- Projection 사용 여부
- 조회 컬럼 최소화 기준
- 처리 시간 측정 방식
- BASIC_LOOP와 비교할 기준

---

### 5.4 벌크 저장 설계

정산 결과를 한 건씩 저장하면 저장 구간이 병목이 될 수 있다.

설계 문서에서 정리할 내용:

- 기존 저장 방식
- `saveAll` 적용 방식
- batch insert 적용 여부
- flush 기준
- 성능 비교 기준

---

### 5.5 인덱스 적용 설계

정산 조건에 맞는 인덱스를 설계해야 한다.

설계 문서에서 정리할 내용:

- 정산일자 조회 조건
- 가맹점별 집계 조건
- 결제/취소 상태 조건
- 중복 정산 검증 조건
- 인덱스 적용 전후 비교 방식

우선 검토할 인덱스:

```txt
payment_date
merchant_id
payment_status
payment_type
payment_date + merchant_id
payment_date + merchant_id + payment_type
settlement_date + merchant_id
```

---

### 5.6 대사 설계

정산 결과와 원천 Payment 데이터를 비교하는 대사 기능은 금융권 포트폴리오에서 매우 중요하다.

설계 문서에서 정리할 내용:

- Payment 기준 집계 금액
- Settlement 기준 집계 금액
- 총 결제금액 일치 여부
- 총 취소금액 일치 여부
- 최종 정산금액 일치 여부
- 불일치 발생 시 처리 방식
- 대사 결과 저장 여부

---

### 5.7 PPT 설명 구조 설계

이 프로젝트는 포트폴리오와 PPT 제작을 전제로 한다.

따라서 기술 구현뿐 아니라 설명 구조도 설계해야 한다.

설계 문서에서 정리할 내용:

- 문제 정의
- 기존 방식의 병목
- 개선 전략
- 성능 비교 결과
- 안정성 확보 방식
- 정합성 검증 방식
- 2단계 증권 실시간 처리 확장 계획
- 면접에서 설명할 핵심 문장

---

### 5.8 2단계 실시간 주식 시세 API 설계

2단계 시작 전, 실시간 시세 API 연동 방식을 설계한다.

설계 문서에서 정리할 내용:

- 사용할 시세 API 후보
- 실시간 시세 수신 방식
- REST 조회와 WebSocket 수신 중 선택 기준
- 인증 정보 관리 방식
- API 실패 시 대체 흐름
- 모의 시세 생성기 사용 여부
- 현재가 데이터 저장 방식
- 현재가를 주문 생성 기준으로 사용하는 방식

권장 방향:

```txt
1순위: 모의 시세 생성기로 현재가 흐름 구현
2순위: 실제 주식 시세 API 연동
```

외부 API 연동은 실패 가능성이 있으므로, 프로젝트 안정성을 위해 모의 시세 생성기를 먼저 구현하고 실제 API는 확장 기능으로 붙인다.

---

### 5.9 2단계 모의 주문 생성기 설계

실제 사용자가 없어도 대량 주문 상황을 만들 수 있도록 모의 주문 생성기를 설계한다.

설계 문서에서 정리할 내용:

- 가상 계좌 생성 기준
- 매수/매도 주문 생성 비율
- 현재가 기준 주문 가격 생성 범위
- 주문 수량 생성 기준
- 초당 주문 수 조절 방식
- 주문 생성 시작/중지 API
- 주문 Queue 적재 방식
- 성능 테스트와 연결할 지표

예시 흐름:

```txt
현재가 수신
→ 현재가 기준 ±1~3% 가격 생성
→ 매수/매도 주문 생성
→ OrderQueue 적재
→ MatchingEngine 처리
```

---

### 5.10 2단계 OrderBook과 Matching Engine 설계

실시간 주문 처리를 위해 Queue와 메모리 기반 OrderBook 구조를 설계한다.

설계 문서에서 정리할 내용:

- 주문 Queue 구조
- 종목별 OrderBook 구조
- 매수 주문 정렬 기준
- 매도 주문 정렬 기준
- 가격 우선·시간 우선 체결 기준
- 부분체결 처리 방식
- 미체결 주문 보관 방식
- 종목별 파티셔닝 여부
- DB 직접 처리 방식과 비교할 성능 기준

권장 자료구조:

```txt
매수 주문: 높은 가격 우선 PriorityQueue
매도 주문: 낮은 가격 우선 PriorityQueue
동일 가격: 먼저 들어온 주문 우선
```

---

### 5.11 2단계 잔고/예수금 정합성 설계

체결이 발생하면 주문 상태, 체결 내역, 보유 수량, 예수금이 함께 변경되어야 한다.

설계 문서에서 정리할 내용:

- 체결 내역 저장 기준
- 매수 주문 상태 변경 기준
- 매도 주문 상태 변경 기준
- 보유 수량 증가/감소 기준
- 예수금 증가/감소 기준
- 잔고 부족 시 주문 거절 정책
- 보유 수량 부족 시 매도 주문 거절 정책
- 부분체결 시 잔여 수량 처리 기준
- 트랜잭션 범위
- 정합성 테스트 기준

체결 시 처리 항목:

```txt
Execution 저장
매수 주문 상태 변경
매도 주문 상태 변경
```

## 10. 설계 결정 기록

중요한 설계 결정은 기록한다.

기록할 내용:

- 어떤 결정을 했는가
- 왜 그 결정을 했는가
- 어떤 대안을 검토했는가
- 선택하지 않은 대안은 왜 제외했는가
- 이 결정이 안정성에 어떤 영향을 주는가
- 이 결정이 정합성에 어떤 영향을 주는가
- 이 결정이 성능에 어떤 영향을 주는가
- 이 결정을 README와 PPT에서 어떻게 설명할 것인가

예시:

```md
## 결정: BASIC_LOOP를 먼저 구현해 성능 개선 전 기준선으로 사용한다.

### 이유

성능 개선 프로젝트에서는 개선 전 기준선이 있어야 GROUP BY, 벌크 저장, 인덱스 적용의 효과를 비교할 수 있다.

### 검토한 대안

- 처음부터 GROUP BY 방식으로 구현
- 처음부터 벌크 저장 적용
- BASIC_LOOP 기준선 구현 후 개선 전략 추가

### 선택

BASIC_LOOP 기준선을 먼저 구현한 뒤, 개선 전략을 단계적으로 추가한다.

### 영향

처리 시간이 느린 기본 방식을 의도적으로 남겨 성능 개선 전후 비교가 가능해진다.

### 발표 포인트

전체 조회와 Java 반복문 집계 방식의 한계를 설명하고, 이후 DB 집계와 인덱스 적용으로 개선한 흐름을 보여줄 수 있다.
```

2단계 증권 실시간 처리 관련 결정 예시는 다음과 같다.

```md
## 결정: 실제 거래소 주문이 아니라 모의 주문·체결 시스템으로 구현한다.

### 이유

실제 거래소 주문은 개인 프로젝트에서 구현하거나 연동할 수 없으며, 포트폴리오 목적상 필요한 것은 실제 주문 전송이 아니라 실시간 주문 처리 구조와 정합성 관리 흐름을 보여주는 것이다.

### 검토한 대안

- 실제 증권사 주문 API 연동
- 실제 시세 API만 연동하고 주문은 모의 생성
- 완전 랜덤 모의 시세와 모의 주문만 사용

### 선택

실제 또는 모의 시세를 기준으로 모의 주문을 생성하고, 자체 Matching Engine에서 체결을 처리한다.

### 영향

외부 시세 데이터를 활용하면서도 주문·체결·잔고 반영 로직은 안전하게 자체 시스템 안에서 구현할 수 있다.

### 발표 포인트

실제 주식 시세 API를 통해 현재가 흐름을 반영하고, 이를 기준으로 모의 주문을 발생시켜 Queue, OrderBook, Matching Engine으로 처리하는 구조를 설명할 수 있다.
```

## 15. 설계 문서 상태

설계 문서는 다음 상태를 사용할 수 있다.

| 상태 | 의미 |
|---|---|
| Draft | 초안 |
| Proposed | 제안됨 |
| Approved | 승인됨 |
| Implementing | 구현 중 |
| Implemented | 구현 완료 |
| Deprecated | 더 이상 사용하지 않음 |
| Planned | 향후 예정 |

문서 상단에 상태를 표시한다.

예시:

```md
# Performance Design

- 상태: Draft
- 작성일: YYYY-MM-DD
- 마지막 수정일: YYYY-MM-DD
- 관련 단계: 1단계 배치
```

2단계 설계 문서 예시:

```md
# Realtime Trading Design

- 상태: Planned
- 작성일: YYYY-MM-DD
- 마지막 수정일: YYYY-MM-DD
- 관련 단계: 2단계 증권 실시간
```

## 16. 에이전트 작업 지침

AI 에이전트가 설계 관련 작업을 할 때는 다음을 따른다.

- 이 색인 문서를 먼저 확인한다.
- 현재 프로젝트 단계가 1단계인지 2단계인지 확인한다.
- 현재는 1단계 배치 성능 개선을 우선한다.
- 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 확정된 기획임을 반영한다.
- 관련 설계 문서가 있는지 확인한다.
- 설계 문서가 있으면 그 내용을 우선한다.
- 설계 문서가 없고 변경 범위가 크면 새 설계 문서를 작성한다.
- 기존 코드와 문서가 다르면 현재 동작하는 코드를 기준으로 판단한다.
- 보안, DB, 배치 흐름, 정합성, 성능 측정에 영향을 주는 변경은 관련 문서도 함께 확인한다.
- 2단계에서 시세 API, 주문 Queue, OrderBook, 체결 정합성, WebSocket에 영향을 주는 변경은 반드시 설계 문서를 먼저 확인한다.
- 설계와 구현이 달라지면 설계 문서를 갱신한다.
- 설계를 임의로 확장하지 않는다.
- 1단계 완료 전 2단계 증권 실시간 기능을 무리하게 구현하지 않는다.
- 복잡한 설계는 대안과 선택 이유를 함께 기록한다.
- 포트폴리오와 PPT에서 설명할 수 있는 구조인지 확인한다.

## 17. 핵심 원칙 요약

- 설계 문서는 구현 전에 방향을 정리하는 문서다.
- 모든 작업에 설계 문서가 필요한 것은 아니다.
- 현재는 1단계 배치 성능 개선을 우선한다.
- 2단계는 실시간 주식 시세 API 기반 모의 주문·체결 처리로 확정한다.
- 안정성, 정합성, 성능 개선이 핵심 키워드다.
- 1단계는 금융권 공통 역량을 보여준다.
- 2단계는 증권사 특화 역량을 보여준다.
- 구조, DB, 배치 흐름, 성능 측정, 대사 기능 변경에는 설계 문서를 작성하는 것이 좋다.
- 실시간 시세 API, 모의 주문 생성, 주문 Queue, OrderBook, 체결 정합성, WebSocket 변경에는 설계 문서를 작성한다.
- 설계 문서는 선택 이유와 대안을 남겨야 한다.
- 실행 계획은 설계 이후에 작성한다.
- 설계와 실제 구현이 달라지면 문서를 갱신한다.
- README와 PPT에서 설명할 수 없는 설계는 포트폴리오 설득력이 약하다.
