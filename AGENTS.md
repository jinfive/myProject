# AGENTS.md

## 프로젝트 개요

이 프로젝트는 대용량 결제 데이터를 활용한 **가맹점 매출 정산 배치 성능 최적화 시스템**이다.

프로젝트의 핵심 목표는 단순히 정산 기능을 구현하는 것이 아니라, 대량의 결제/취소 데이터를 일별로 정산하는 배치 처리 과정에서 발생하는 성능 병목을 확인하고, 쿼리 튜닝, 인덱싱, 벌크 저장, 처리 방식 개선 등을 통해 성능을 개선하는 과정을 포트폴리오로 정리하는 것이다.

정산 담당자는 프론트 화면에서 정산 배치를 실행하고, 정산 결과와 배치 실행 시간, 처리 건수, 성공/실패 건수, 적용된 처리 전략을 확인할 수 있어야 한다.

이 프로젝트는 **성능 개선 중심 프로젝트**다.

## 개발 원칙

- 기능은 Issue 단위로 작게 구현한다.
- main 브랜치에는 직접 작업하지 않는다.
- develop 브랜치에서 기능을 통합한다.
- 기능 개발은 feat/{issue-number}-{feature-name} 브랜치에서 진행한다.
- 각 기능 구현 후 Pull Request를 통해 develop에 병합한다.
- 프로젝트 목적과 관련 없는 기능은 추가하지 않는다.
- 회원가입, 로그인, 권한 관리, 장바구니, 주문 화면, 복잡한 관리자 기능은 구현하지 않는다.
- 핵심은 정산 배치 성능 개선이므로 CRUD 기능은 최소화한다.
- 화면은 정산 담당자가 결과와 성능 지표를 확인할 수 있는 수준으로 구현한다.
- 기능을 많이 추가하기보다 성능 개선 흐름을 깊게 구현한다.
- 성능 개선 전후를 비교할 수 있도록 처리 전략을 명확히 구분한다.

## 도메인 범위

기본 도메인은 다음과 같다.

- Merchant: 가맹점
- Payment: 결제/취소 데이터
- Settlement: 정산 결과
- BatchJobHistory: 배치 실행 이력
- SettlementError: 정산 실패 이력

복잡한 상품, 회원, 로그인, 권한 기능은 구현하지 않는다.

이 프로젝트에서 결제 데이터는 실제 사용자가 결제하는 방식이 아니라, 성능 테스트와 정산 배치 처리를 위해 더미 데이터로 대량 생성한다.

## 핵심 데이터 흐름

정산 배치 흐름은 다음과 같다.

1. 대용량 결제/취소 데이터 생성
2. 특정 정산일자 기준으로 결제 데이터 조회
3. 가맹점별 결제금액 집계
4. 가맹점별 취소금액 집계
5. 가맹점별 수수료 계산
6. 최종 정산금액 계산
7. 정산 결과 저장
8. 배치 실행 시간 및 처리 건수 저장
9. 프론트 화면에서 정산 결과와 성능 지표 조회
10. 개선 전후 처리 시간 비교

## 정산 규칙

- 결제 완료 데이터는 매출에 포함한다.
- 취소 완료 데이터는 매출에서 차감한다.
- 가맹점별 수수료율을 적용해 수수료를 계산한다.
- 순매출 = 총 결제금액 - 총 취소금액
- 수수료 = 순매출 × 가맹점 수수료율
- 정산금액 = 순매출 - 수수료
- 같은 가맹점과 같은 정산일자는 중복 정산되지 않아야 한다.
- 정산 대상 데이터가 없으면 불필요한 정산 결과를 생성하지 않는다.
- 정산 처리 중 오류가 발생하면 실패 이력에 원인을 기록한다.

## 대용량 더미 데이터 기준

더미 데이터는 성능 개선 시나리오를 만들 수 있도록 충분히 많이 생성한다.

기본 기준은 다음과 같다.

- 가맹점 수: 최소 100개 이상
- 결제/취소 데이터 수: 최소 100,000건 이상
- 가능하면 500,000건 또는 1,000,000건까지 확장 가능하게 구성
- 여러 날짜의 거래 데이터 포함
- 특정 정산일자에 데이터가 몰려 있는 케이스 포함
- 결제 상태와 취소 상태가 모두 포함
- 각 가맹점은 서로 다른 수수료율을 가짐
- 더미 데이터 생성 시간도 로그로 확인 가능해야 함

대량 데이터 생성은 애플리케이션 실행 시 자동 생성하거나, 별도의 데이터 생성 API 또는 배치로 실행할 수 있게 한다.

## 처리 전략

성능 개선 전후 비교를 위해 정산 배치 처리 전략을 구분한다.

### BASIC_LOOP

기본 구현 방식이다.

- 특정 날짜의 Payment 데이터를 전체 조회한다.
- Java 반복문으로 가맹점별 결제금액과 취소금액을 집계한다.
- 수수료와 정산금액을 계산한다.
- Settlement를 저장한다.
- 처리 시간을 BatchJobHistory에 기록한다.

이 방식은 성능 개선 전 기준점으로 사용한다.

### GROUP_BY_QUERY

쿼리 개선 방식이다.

- Payment 전체 조회를 줄인다.
- DB에서 GROUP BY로 가맹점별 결제금액과 취소금액을 집계한다.
- 필요한 집계 결과만 Service 계층으로 가져온다.
- Settlement를 저장한다.
- BASIC_LOOP 방식과 처리 시간을 비교한다.

### BULK_SAVE

저장 성능 개선 방식이다.

- Settlement를 한 건씩 저장하지 않는다.
- 정산 결과를 모아 한 번에 저장한다.
- 반복적인 save 호출을 줄인다.
- 처리 시간을 이전 방식과 비교한다.

### INDEX_APPLIED

인덱스 적용 후 비교 방식이다.

- 정산 조건에 자주 사용되는 컬럼에 인덱스를 적용한다.
- 같은 데이터 건수와 같은 정산일자 기준으로 처리 시간을 비교한다.
- 인덱스 적용 전후 성능 차이를 README에 정리한다.

## 성능 개선 시나리오

README에 성능 개선 과정을 반드시 기록한다.

성능 개선 시나리오는 다음 흐름으로 작성한다.

### 1단계: 기본 구현

처음에는 단순한 방식으로 정산 배치를 구현한다.

예시:

- 전체 결제 데이터를 조회
- 애플리케이션 메모리에서 가맹점별 그룹화
- 반복문으로 정산금액 계산
- 정산 결과 저장

이 단계에서는 대용량 데이터 처리 시 시간이 오래 걸리는 문제를 확인한다.

### 2단계: 병목 확인

다음 정보를 기록한다.

- 데이터 건수
- 정산 대상 날짜
- 처리 전략
- 처리 시간
- 조회 쿼리 수
- 느린 구간
- 메모리 사용 문제 여부

### 3단계: 쿼리 튜닝

가능한 경우 DB에서 직접 집계하도록 개선한다.

예시:

- GROUP BY로 가맹점별 결제금액/취소금액 집계
- 불필요한 전체 데이터 조회 제거
- 필요한 컬럼만 조회
- N+1 쿼리 발생 여부 확인

### 4단계: 인덱스 적용

정산 조건에 자주 사용되는 컬럼에 인덱스를 적용한다.

우선 고려할 인덱스는 다음과 같다.

- payment_date
- merchant_id
- payment_status
- payment_type
- merchant_id + payment_date
- payment_date + merchant_id + payment_type
- settlement_date
- merchant_id + settlement_date

적용 전후 실행 시간을 비교한다.

### 5단계: 저장 방식 개선

필요하면 다음 방식도 적용한다.

- save 반복 줄이기
- saveAll 사용
- 벌크 저장 적용
- 불필요한 flush 줄이기
- 통계성 쿼리 분리

### 6단계: 개선 결과 정리

README에 다음 형식으로 정리한다.

| 전략 | 데이터 건수 | 처리 시간 | 개선 내용 |
|---|---:|---:|---|
| BASIC_LOOP | 100,000건 | 측정값 입력 | 전체 조회 후 Java 반복문 집계 |
| GROUP_BY_QUERY | 100,000건 | 측정값 입력 | DB GROUP BY 집계 |
| BULK_SAVE | 100,000건 | 측정값 입력 | 정산 결과 일괄 저장 |
| INDEX_APPLIED | 100,000건 | 측정값 입력 | 조회 조건 인덱스 적용 |

성능 수치는 로컬 환경에 따라 달라질 수 있으므로, 절대 성능보다 동일 환경에서의 개선 전후 비교를 목적으로 기록한다.

## 트랜잭션 처리 기준

정산 배치 처리 과정에서는 데이터 정합성을 위해 트랜잭션을 적용한다.

다음 작업은 하나의 흐름으로 관리한다.

- 정산 대상 데이터 조회
- 정산 결과 계산
- 정산 결과 저장
- 배치 실행 이력 저장
- 실패 이력 저장

중복 정산이 발생하지 않도록 같은 가맹점과 같은 정산일자 기준으로 검증한다.

기본 방향은 다음과 같다.

- 전체 배치 실행 이력은 항상 남긴다.
- 특정 가맹점 정산 실패 시 SettlementError에 실패 원인을 저장한다.
- 성공한 가맹점 정산 결과는 저장한다.
- 실패한 가맹점은 실패 이력으로 관리한다.
- 같은 가맹점과 같은 정산일자에 대해 중복 정산 결과가 생성되지 않도록 한다.

## 코드 작성 기준

- Java 17과 Spring Boot 기준으로 작성한다.
- Controller, Service, Repository, Entity 계층을 명확히 분리한다.
- 비즈니스 로직은 Controller에 두지 않는다.
- 정산 계산 로직은 Service 계층에 둔다.
- 금액 계산에는 double/float를 사용하지 말고 BigDecimal을 사용한다.
- 상태값은 문자열보다 Enum을 우선 사용한다.
- 처리 전략은 Enum으로 관리한다.
- 중복 정산 방지는 애플리케이션 검증과 DB 제약조건을 함께 고려한다.
- 중요한 처리 흐름에는 로그를 남긴다.
- 예외 메시지는 원인을 이해할 수 있게 작성한다.
- 성능 측정을 위해 배치 시작 시간과 종료 시간을 기록한다.
- 프로젝트 목적과 관련 없는 기능은 추가하지 않는다.
- 대량 데이터 처리 시 불필요한 전체 객체 로딩을 줄이는 방향으로 개선한다.

기능 구현 후 가능한 범위에서 테스트 코드를 작성한다.

우선순위는 다음과 같다.

1. 정산 금액 계산 테스트
2. 취소 금액 반영 테스트

4. 중복 정산 방지 테스트
5. 정산 대상 데이터 없음 처리 테스트
6. 배치 실행 이력 저장 테스트
7. 대용량 데이터 기준 정산 실행 시간 측정 테스트
8. BASIC_LOOP와 GROUP_BY_QUERY 성능 비교 테스트
9. 인덱스 적용 전후 성능 비교 테스트

성능 테스트는 로컬 환경 영향을 받으므로 절대적인 수치보다 동일 조건에서의 개선 전후 비교를 목적으로 한다.

테스트 작성이 어렵다면 최소한 실행 가능한 API 요청 예시와 예상 결과를 README에 남긴다.

## Codex 작업 방식

작업을 시작하기 전에 현재 브랜치와 변경 파일을 확인한다.

작업은 한 번에 하나의 기능만 진행한다.

새 기능을 만들 때는 다음 순서로 진행한다.

1. 요구사항을 짧게 요약한다.
2. 필요한 파일을 확인한다.
3. 구현 계획을 간단히 세운다.
4. 코드를 수정한다.
5. 실행 또는 테스트 가능한 방법을 제시한다.
6. 변경 내용을 요약한다.
7. 다음 작업 후보를 제안한다.

불필요하게 큰 구조 변경은 하지 않는다.
새로운 라이브러리는 꼭 필요한 경우에만 추가한다.
기존 코드 스타일이 있다면 기존 스타일을 따른다.

## Codex에게 특히 강조할 사항

이 프로젝트의 핵심은 정산 기능 자체가 아니라, **대용량 정산 배치 처리와 성능 최적화 과정**이다.

Codex는 다음 기능을 임의로 추가하지 않는다.

- 회원가입
- 로그인
- 권한 관리
- 복잡한 상품 관리
- 장바구니
- 주문 화면
- 불필요한 CRUD 화면
- 프로젝트 목적과 관련 없는 기능

우선순위는 다음과 같다.

1. 대용량 더미 데이터 생성
2. 기본 정산 배치 구현
3. 배치 실행 이력 저장
4. 처리 시간 측정
5. 쿼리 튜닝
6. 인덱싱 적용
7. 벌크 저장 적용
8. 정산 대시보드 구현
9. 성능 테스트 작성
10. 성능 개선 전후 비교
11. README 정리

항상 “대용량 데이터를 배치로 정산할 때 어떻게 성능을 개선했는가”를 설명할 수 있도록 구현한다.
